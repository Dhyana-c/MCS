# 设计：聚焦候选集选择

## 1. 实验证据（驱动本设计）

同图 `dschat_full_16k`、同 200 非-null query、同 lexical `doc_rerank`，agent（deepseek ReAct）vs 框架（固定 BFS + 宽召回 select_facts）。截至分析时样本 ~50–74 题（跑批进行中，最终数见 `bench/multihop_rag/reports/agent_vs_framework.md`）：

| 指标 | agent | 框架 | 解读 |
|---|---|---|---|
| reached（≥1 篇 gold，二元） | ~0.94 | ~0.94 | 导航/召回打平 |
| recall@∞（完整召回率） | ~0.86 | ~0.80 | agent 微胜（多捞回次要 gold 篇） |
| hit@10 | ~0.80 | ~0.72 | agent 胜 |
| recall@10 | ~0.52 | ~0.41 | agent 胜 |
| **gold 首篇平均名次（都召回到的题）** | **~7** | **~18** | **核心：框架把 gold 埋得深** |
| LLM 调用/题 | ~25 | ~21 | 调用数仅多 ~20% |
| token/题 | ~150K | 较低 | agent 溢价主要在 ReAct 上下文膨胀 |

**结论**：差距主体在**排序**，根因是框架候选集**不够聚焦**（宽召回 BFS 收入大量旁支，gold 被噪声稀释）。agent 赢在 LLM 引导的聚焦；它的失败则多来自**探爆**（如 645 节点→gold@210）或 **deepseek 早停/用参数知识答**（某题 0 工具直接作答）。

**非确定性警告**：agent 同题两次跑可 win↔lose 翻面（如 `ed2bb96d847b`、`6aa1c570910a` 在 15 例研究里 reached，全量跑里因探爆失败）。逐类型小样本数字仅供参考，勿当铁数。

## 2. deepseek 的硬约束（贯穿两方案）

见 `docs/select_facts_model_differences.md`：deepseek 在**受限输出下的精细判别 + 指令遵循**上弱（曾 75.7% 返裸 `[]`、可空就崩、易早停），强模型（GLM）能守。推论：

- **任何"让 deepseek 在探索期自判够了就停"的设计都危险**——它会早停丢召回。框架的宽召回 prompt 就是为治这个而存在。
- 安全的做法是把"探索充分"与"输出精筛"**彻底分开**：探索期只管宽、不准停；聚焦判断放到**事后**（方案 A），或用**可回访兜底**（方案 B）。

## 3. 方案 A：宽召回 + 事后充分性裁剪（推荐，deepseek 优先）

### 流程
1. `_traverse` 照常宽探，得 `accumulated`（不改终止逻辑，不早停）。
2. 新增 **充分性裁剪**：一次 LLM 调用，输入查询 + accumulated 的节点视图，输出"足以回答查询的最小聚焦子集"（节点 id 列表）。
3. **只把聚焦子集喂 doc_rerank**；同时保留 accumulated 全集。
4. **兜底**：裁剪结果为空或显著少于阈值（如 < gold 期望量 / < K）→ 回退用 accumulated 全集，杜绝剪过头。

### 落点
- 作为 `POSTPROCESS` 或 `TRIM` 插件（`mcs/core/query_engine.py` 遍历收尾后调用），不侵入遍历循环。
- 新增 prompt bundle `mcs/prompts/select_sufficient.py`（SYSTEM/USER + parse + 兜底归一）。

### 为何安全
- 探索期从不早停 → 不触发 deepseek 的早停弱项。
- 裁剪剪错有全集兜底 → 召回不可能因裁剪下降。
- 改动小、可灰度（开关：裁剪 on/off 对照）。

### 风险
- 多一次 LLM 调用（每题 +1，相对 ~21 次可忽略）。
- deepseek 裁剪判断仍不完美 → 靠兜底阈值 + 全集回退控制。

## 4. 方案 B：有界可回访的 LLM 引导遍历（备选 / Phase 2）

### 与现状的差异
| | 现状 `_traverse` | 方案 B |
|---|---|---|
| 终止 | `max_rounds` + frontier 耗尽 + token 预算 | **`max_steps` 步数预算** + token 预算 |
| 去重 | `visited` 永久禁访 | **去掉永久禁访，允许回访** |
| 扩展选择 | 按层批量展开 frontier | **LLM 每步选展开谁 / 忽略谁** |

### 进度保证（关键设计点）
允许回访 → 必须防"deepseek 在热 hub 上反复回访、空烧步数预算"：
- **回访门**：某节点只有在"自上次对它裁决后 `accumulated` 发生变化"时才准再次展开（有新上下文才回头）。
- 或：对回访加软惩罚 / 限制单节点回访次数上限。
- 无进度保证则方案 B 会自杀（步数耗尽却没推进）。

### 为何对 deepseek 反而友好
- **可回访 = 过激剪枝的回收路**：deepseek 这步嫌某点不重要跳过，后面 accumulated 变了又需要，可回来——现有 `visited` 一旦跳过永不回头，剪错即丢 gold。
- 源头剪枝（忽略不重要点）直接治"探爆→埋 gold"。

### 本质
把 agent 制胜的"LLM 聚焦引导 + 有界 + 可回访"做成框架原语，**丢掉 ReAct 的对话上下文膨胀**（agent token 溢价的唯一来源）。是比方案 A 更忠实、更省成本的"移植 agent"，但改动大、押 deepseek 重要性判断（有回访兜底）。

### 风险
- 重写遍历终止/去重，触及核心不变量路径，需充分测试。
- 押注 deepseek 的"重要性"判断；上线前需 deepseek 实测，或先用 GLM 验证可行性。

## 5. 开放问题（待定）

1. **步数预算单位**（方案 B）：按"节点展开次数"还是"LLM 调用次数"计？默认值多少？
2. **充分性判据**（方案 A）：让 LLM 对照查询的子实体/子问题逐条确认"已覆盖"，还是整体判断？前者更稳但更贵。
3. **裁剪兜底阈值**（方案 A）：聚焦子集下限取多少（绝对 K / 相对 accumulated 比例 / gold 期望量）？
4. **是否叠加**：B + A 同时上，还是二选一先验证？
5. **模型分流**：deepseek 走 A，GLM 是否直接走 B（它守得住重要性判断）？

## 6. 验收标准

- 同口径对照框架基线：**reached / recall@∞ 不下降**（聚焦不丢召回）+ **gold 平均名次下降、hit@10 / recall@10 上升**。
- 方案 A：裁剪 on/off 对照，证明提升来自裁剪；兜底回退路径有测试覆盖（空/超剪都不丢召回）。
- 方案 B：进度保证有测试（构造热 hub，证明不空烧预算）；`accumulated ≤ T` 不变量仍守。
- 全量单测绿；deepseek 实测剪枝/回访行为符合预期（不早停丢召回）。
