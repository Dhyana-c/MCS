# unified-graph-schema — Design（决策记录）

> 模型的**完整、权威定义**见 [`docs/graph-model-design.md`](../../../docs/graph-model-design.md)。本文**不重复定义**数据结构与算法，只记录**关键决策与其理由**、决策演进、与现状的差异 / 迁移、盲区与未决问题。

---

## 1. 关键决策（最终）

### D1：谓词落点——事实即节点（能力轴判据）

事实统一为命题节点、谓词落 `content`，取消 label 事实边。**判据是能力轴，不是性能轴**：未来要 `事件 → 事实` 背书与 `事实 ↔ 事实` 互斥，而**边连不了边**——事实必须是一等节点才表达得了。一个表达不了必需能力的模型，QA 再好也出局；QA 回归只用来量实现风险（召回缺口），不作 go/no-go。与 [[memory-agent-fact-node-model]] 的"fact 超边节点"同源。

### D2：节点 4 类，hub 仅标记

节点按结构行为分 **概念 / 事实 / 事件 / source** 四类（不引入领域 type）。`hub` 降为**标记**（反查 / 可观测），无算法含义、非节点类——它只是恰好成为组织中心的普通节点。

### D3：边极简——仅 关联 + 互斥，无独立层级边

边只保留 `关联`（结构基础边）+ `互斥`（当前唯一语义类型）。**没有"层级"边**：组织层级是聚类的产物，用关联边 + hub 标记表达，不需要独立边类型。开放谓词不进边（落事实 content），新增语义类型走登记制、谨慎增加。

### D4：核心 / 事件双层，核心不反查事件

核心图（概念 + 事实）有界；事件层单向绑入、**核心不反查事件**（尤其"用户 / 我"——否则最热节点把全部事件漏回核心、撑爆活跃视图）。这条是有界的**命根**。[[memory-agent-fact-node-model]] §0.1 命门（事实变节点后概念邻域更易超 T）由此**由构造化解**：把事件移出核心后，核心度数由稳定命题数决定（天然小）。

### D5：语义归 LLM，结构归规则

概念 / 事实靠 LLM 语义抽取；**事件 / source 由规则入库、不经 LLM**。这把"命题 vs 事件"的判断从 LLM 手里**拿掉**了——事件是输入自带的结构、直接存，转述的"三年前 X"则被抽成带时间属性的事实、不污染时间轴。**写入复用 read**：抽取前先检索已有节点对齐，避免重复。

### D6：守门挂在改图操作上 + 边吸收

守门不专属写入——**写 / 连边 / 合并 / 读修复**后都过守门，超 T 必聚类裂变（合并也会撑大邻域，故同样过守门）。新组织中心生成后做**边吸收**（子节点 ⊇ 中心成员的节点改连中心，复用 hub、减边）。有界对**一切 mutation** 成立。

### D7：质量收敛——读写触发 + read-repair + 维护兜底

去重 / 合并由读写共同触发、多次机会：创建对齐、之后被写 / 读触及（**read-repair**，读时撞见重名当场可见、同义搭 ③ 的 LLM 顺带判）、聚类合并；长尾残留靠**可选后台维护**兜。收敛是 best-effort 最终收敛，**同名 ≠ 同义**需消歧，事实不走合并。

## 2. 决策演进（诚实记录）

本 change 经历了一次方向翻转，记录以免反复：

- **第一轮**：保留 `role(concept/hub/attribute)` + 用"封闭可扩展 `type` 枚举"替代 `kind`+`label` + `event/source/assertion` 三类物化点（靠 `extensions` 标记、不新增 role）。曾**明确否决** 4 类节点模型，理由是"概念 / 事实 / 事件算法行为一致"。
- **翻案**：能力需求（事件背书、事实互斥）要求事实是一等节点 → 谓词落点；而事件移出核心后行为与概念**不再一致** → 4 类节点成立、否决的前提失效。`assertion` 物化点被"互斥 = 事实间边"取代；`type` 枚举收敛为仅 `关联 / 互斥`；新增"双层 + 守门=mutation + 边吸收 + read-repair + W=S+T+R"。
- 结论以 `docs/graph-model-design.md`（v0.1）为准。

## 3. 与现状的差异与迁移（爆炸半径）

本 change 是**彻底重构**，须显式认领以下 deltas（不得"待实现时定"）：

| 对象 | 动作 |
|------|------|
| `dual-edge-model`（label 事实边） | **REMOVE** |
| `attribute-node-model` 的 `relation_model` 开关 | **REMOVE**（其属性节点 + 关联边思路作统一模型的底） |
| `relation_model` 双模式 | 删除 |
| `entities-package` | `Edge`（`kind`/`label`→`type`，仅 关联/互斥）；`Node` 加 `node_class` |
| `subgraph-bounding` | 守门挂 mutation + 边吸收；`type` 不计 token |
| `write-pipeline` / `query-pipeline` | 规则入库 + 复用 read；核心 BFS + read-repair |
| 宪法 `CLAUDE.md` | 重写 `relation_model` 不变量与两条铁律措辞（冲突先改宪法） |

现有 db（multihop-rag 产出）建议**新库重建**（评测库重建更干净），不写迁移脚本。

## 4. 盲区与代价

完整见 `docs/graph-model-design.md §7`。要点：

1. **抽取质量是唯一真正脆的点**：LLM 抽概念 / 事实的准确率（漏抽、概念 vs 事实误判、谓词归纳）。事件 / source 规则入库已消除"命题 vs 事件"误判与时间轴污染。须用记忆样本量化（含转述嵌套）。
2. **去重依赖检索召回 + 同名歧义**：异名同义漏召回 → 暂存重复待兜底；同名未必同义，消歧有误判风险。
3. **事件全量查新近偏置**：靠关键事件蒸馏成核心事实兜。
4. **入边 / 关联截断丢冷门**；**n 元互斥**当前两两连边、规模大偏冗余。

2. 登记制实现形态。
3. 事件 / source 的 `extensions` 字段约定（timestamp / 帧 / 参与者 / source_type）。
4. 后台维护扫描的触发与算力预算。
5. 迁移顺序：先核心机制单场景验，还是先搭三场景评测。

这些不在本 change 冻结。
