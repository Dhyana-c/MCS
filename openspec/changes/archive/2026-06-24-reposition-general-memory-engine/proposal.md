## Why

项目文档当前以 **「核心赌注：知识有足够的局部性」+「面向单一领域」** 自我限定——把 MCS 框定为一个只在"可教 / 单领域"知识上成立的"赌注型"系统。这与项目实际目标（一个**完整的通用记忆引擎**）不符。更关键的是，这套表述把"局部性"错误地描述成对**外部数据**的*假设*；而架构上，局部性是系统靠**核心不变量 + 守门 + 聚类裂变主动维持**的*保证*——项目宪法 `CLAUDE.md` 本就按"维持"而非"假设"来表述，只是 `README` / `architecture` / `faq` 三份用户文档还停留在旧的"赌注"口径上。现在把文档对齐到宪法的真实口径，并解除不必要的领域自限。

## What Changes

- **重定位**：`README.md` / `docs/architecture.md` / `docs/faq.md` / `docs/INDEX.md` 中"面向单一领域的记忆系统"→"**完整的通用记忆引擎**"，删除"对可教领域最成立 / 对跨领域、矛盾常态化知识发紧"的领域限定。
- **局部性改述为系统保证**：把"**核心赌注：知识有足够的局部性**（假设外部知识就近）"改述为"**系统通过核心不变量 + 守门 + 聚类裂变主动维持语义局部性**（任意节点活跃双向视图渲染 token ≤ T）"——**维持而非假设**。删除"赌注 / 假设"措辞。
- **README 结构调整**：「核心赌注」节 → 「核心定位」节（内容由"赌注 + 领域限定"改为"通用引擎定位 + 核心不变量指针"）。
- **向量检索立场不动**：保留"默认不依赖 embedding / 向量检索"的结论原样；`docs/faq.md` 中「为什么不用向量检索」一句仅把"核心赌注是局部性"的**局部性措辞**改述为"LLM 直接读有界活跃视图"，**结论不变**。
- **无代码 / API / 依赖变更**——架构完全不动（仍沿关联边 BFS、默认不引入向量）。
