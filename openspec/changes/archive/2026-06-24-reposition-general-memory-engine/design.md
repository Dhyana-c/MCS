## Context

MCS 的用户文档（`README.md` / `docs/architecture.md` / `docs/faq.md`）当前以 **「核心赌注：知识有足够的局部性」+「面向单一领域」** 作为开篇定位，把系统框定为一个只在"可教 / 单领域"知识上成立的赌注型记忆系统。这带来两个问题：

1. **定位与目标不符**：项目实际目标是**完整的通用记忆引擎**，而文档开篇就自我设限（"对跨领域综合、强语境依赖、矛盾常态化的知识会发紧"）。
2. **口径错误**：文档把"局部性"表述为对**外部数据**的*假设*（赌注）；而架构上，局部性是系统靠**核心不变量 + 守门 + 聚类裂变**主动*维持*的属性——项目宪法 `CLAUDE.md` 本就按"维持"口径书写（全文无"赌注/局部性假设"措辞），用户文档却停留在旧口径。

## Goals / Non-Goals

**Goals:**

- 把定位从"面向单一领域的赌注型记忆系统"重写为"**完整的通用记忆引擎**"。
- 把"局部性"从"假设 / 赌注"改述为"系统主动维持的保证"（**维持而非假设**），与宪法口径对齐。
- README「核心赌注」节 → 「核心定位」节，并同步更新 `doc-hierarchy` spec 的对应 SHALL 文本。
- 保留"默认不依赖 embedding / 向量检索"结论（仅改其局部性措辞依据）。

**Non-Goals:**

- **不动架构**：不改 BFS 召回链路、不引入向量 / 混合检索、不改核心不变量与守门逻辑——纯文档变更。
- **不动 `CLAUDE.md`**：宪法已按"维持"口径书写，无需改动。
- **不动代码**：`mcs/` / `mcs_mcp/` / `mcs_agent/` / `bench/` 零改动。
- 不为 README 文案内容（具体定位措辞）新增 spec 契约——doc-hierarchy 只约束结构 / 完整性，不约束营销措辞。

## Decisions

### D1：局部性"维持而非假设"（改述，不删除）

**选择**：把"赌注 / 假设知识有局部性"改述为"系统通过核心不变量 + 守门 + 聚类裂变主动维持语义局部性（任意节点活跃双向视图渲染 token ≤ T）"。

**理由**：现有架构的召回（沿关联边 BFS）确实依赖邻域语义内聚——但这份内聚不是赌外部数据天然就近，而是**守门 + 聚类裂变在每次改图后强制收敛出来的**（宪法「守门」「聚类裂变」节）。改述为"保证"既解除领域自限，又与架构零矛盾、解释力更强。

**备选**：
- *直接删除不替代*：召回机制失去解释，BFS / 无向量架构衔接变弱——否决。
- *保留但降格*：仍隐含"外部数据有局部性"，与"通用引擎"定位冲突——否决。

### D2：README 节改名（「核心赌注」→「核心定位」），不删节

**选择**：保留 README 的定位节，仅改名 + 重写内容。

**理由**：`doc-hierarchy` spec 的「README 精简」Requirement SHALL 要求 README 含定位类章节；删节会违反 spec，改名则保留开源惯例结构、只去掉"赌注"框架。新内容 = 通用引擎定位 + 核心不变量指针。

### D3：向量立场不动，`faq.md:51` 最小改写

**选择**：保留"默认不用向量"结论，仅把"核心赌注是局部性"改述为"LLM 直接读有界活跃视图"。

**精确改写**：

```
现状 (faq.md:51):
  MCS 的核心赌注是知识有局部性，不需要向量相似度兜底。大模型直接阅读
  "装得下的局部子图"做判断，比向量相似度更准确（尤其在关系发现、聚类场景）。

目标:
  MCS 默认不依赖向量相似度兜底——大模型直接阅读"装得下的有界活跃双向视图"
  做判断（关系发现、聚类、召回），靠核心不变量保证局部视野装得下、比向量
  相似度更准确（尤其在关系发现、聚类场景）。
```

**理由**：用户明确"向量立场暂不触碰"。该句同时含局部性（D1 目标）与向量（用户划禁区），最小改写：换掉局部性措辞、保留"不需要向量兜底"结论与"更准确"判断——两全。

### D4：零代码改动

**理由**：定位 / 框架表述与架构实现正交——BFS 召回、默认无向量、核心不变量都不依赖"把局部性叫赌注还是保证"。故重定位是纯文档工作，不触发回归。

## Risks / Trade-offs

- **[口径断裂]** `faq.md:51` 局部性与向量耦合 → D3 最小改写保结论、换依据，已消除断裂。
- **[spec 漂移]** doc-hierarchy 仍要求"核心赌注"节名 → spec delta 同步为"核心定位"。
- **[宪法 vs 文档漂移]** → 宪法本就干净，本次让文档追上宪法，无新增漂移。
- **[过度承诺风险]** "通用记忆引擎"定位不应暗示系统能处理任意无结构数据——README「核心定位」节须诚实说明其仍以**语义结构化记忆**为长（不夸大），design 在 tasks 中以"诚实措辞"约束。
