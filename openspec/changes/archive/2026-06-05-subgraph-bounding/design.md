## Context

量化诊断（418 query 文档级评测）给出硬数据：**MCS 的语义层在空转**——平均只调 1.1 次大模型/query，召回几乎全靠 `alias_entry` 词法种子。零成本验证显示每 query 命中 ~1200 个种子、其中 30% 孤立、平均度 1.73。

成因是图的**纵向抽象层缺失**：
- `decide_hub`（LLM）**设计了**「提拔现有枢纽 / 合成新上位概念」，但执行它的 `fanout_reducer` **只标 `role="hub"` / 记一句 `synthetic_hub_summary` 备注，不真正建节点、不重组边**（注释："图手术留待未来完善"）；
- 触发阈值硬编码 `fanout_threshold=12`，平均度 1.32 下几乎不触发（4380 节点仅 32 个名义 hub）；
- 查询时 ~1200 个扁平种子被 `priority_trim` 按 token 预算**暴力截断**，从未归纳成分层「种子图」；
- 底座 `TokenBudget.estimate = len//2` 对英文**高估约 2.3×**，让所有 token-aware 判据失真。

**约束**：提中间概念**必须 LLM 参与**（语义归纳，非纯聚类）。**当前是测试阶段**：直接重建图验证机制，不做成本优化、不做 offline 手术。

## Goals / Non-Goals

**Goals:**
- 确立统一原则：任何节点（含查询虚拟根）的子图 > 上下文容量 → 用 LLM 提中间概念节点分层。
- token 估计修准 + fanout 阈值 token-aware（由窗口/预算推导，16k → ~32）。
- 真正落地中间概念提取：`fanout_reducer` 建中间节点 + 重组边。
- 查询「虚拟根 + 种子图」：种子过多时递归/分层 LLM 归纳，替掉暴力 trim。
- 验证语义层是否真正展开（中间概念分层是否让遍历动起来）。

**Non-Goals:**
- 不一次定死所有参数（递归归纳机制先小规模实验）。
- 不做有向/带类型边、跨文档补边（留 `graph-construction-quality`）。
- **测试阶段不做 LLM 成本优化（批次/缓存等）、不做 offline 图手术**——直接重建图验证机制；成本与 offline 增量留后期。
- 不动已归档的 reranker / 持久化。

## Decisions

### D1: 统一原则 + 虚拟根
「任何节点的子图（邻域渲染）≤ 上下文容量，超则提中间概念」。查询种子集 = 一个**虚拟根节点**的子图——于是建图（普通节点邻域溢出）与查询（种子集过大）**共用同一套 fanout reduction**，只是入口不同。**理由**：一套机制覆盖两个场景，概念统一、实现复用。

### D2: 修 token 估计（底座）
`len//2` → 更准的估法。**首版**用廉价经验式（英文 ≈ `len/4` 或 `词数×1.3`；CJK 仍近似 `len`），并把估计收敛到 `TokenBudget.estimate` 单一入口，**预留接口**后续换真分词器（如 tiktoken）。**理由**：阈值/trim/子图预算全依赖它，必须先修；但不想首版就引入重依赖。

### D3: token-aware 的 fanout 阈值
触发不再是固定邻居数，而是「邻域**渲染 token** 累计 > 可用窗口」时触发（等价于 ≈ `W ÷ 每节点渲染 token`，16k → ~32）。给一个 floor（如 16）避免极端。**理由**：阈值应随上下文窗口与节点大小自适应，12 是拍脑袋且在稀疏图上从不触发。

### D4: 中间概念节点真正落地
`fanout_reducer` 在 `decide_hub` 归纳后：
- `hub_id` 有值 → 提拔该现有节点为中间枢纽，并把同组成员的连接收敛到它（星型重组）；
- `hub_id=None` + `synthetic_hub_summary` → **真正新建**一个中间概念节点（`role="hub"`，content=摘要），把成员挂到它下面、重组边。

**理由**：这是补完「图手术」那半截——只有真正建节点+重组边，图才有纵向层次，hub_fallback 的自顶向下导航才有东西可导。

### D5: 虚拟根 + 种子图的递归归纳
阶段② 种子集 > 阈值时：**自底向上分批**（每批 ≤ 阈值）`decide_hub` 归纳出中间概念 → 若中间概念数仍 > 阈值则**递归**，直到虚拟根的直接子集 ≤ 阈值。产出一棵分层「种子图」，遍历从中间概念入手、按需下钻。**理由**：1200 种子塞不进一次 LLM；递归归纳把扁平大集合收敛成 `O(log)` 层次。

## Risks / Trade-offs

- **[LLM 成本上升]** 提中间概念是 LLM 调用，LLM/query 会升 → 测试阶段先**接受**、聚焦验证机制是否让语义层展开；成本优化留后期。
- **[归纳质量]** LLM 提的中间概念不准/太泛 → 缓解：`decide_hub` 优先提拔现有节点、prompt 调优、对照实验验证检索增量。
- **[中间概念污染检索]** 提出的 hub 概念被当种子召回但不相关 → 缓解：中间概念主要用于**组织/导航**，检索打分时可降权或只作下钻入口。
- **[动了既有 baseline]** 改种子定位可能影响现有评测基线 → 缓解：opt-in 开关 + 三方对照（关/建图侧/查询侧）。
- **[token 估计精度]** 经验式仍有偏差 → 缓解：单一入口 + 预留真分词器接口；阈值留 floor 容错。

## Open Questions

- 递归归纳的**分批策略**具体怎么定（按图邻接分组？按相似度？）。
- 中间概念节点在**检索时如何被使用**：作种子参与召回 / 只作下钻导航 / 打分降权？
- token 估计精度到什么程度够用——经验式 vs 真分词器的收益/成本。
- 阈值（floor、窗口占比）、递归深度的取值——需小图实验扫一遍。
