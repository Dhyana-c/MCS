## Context

这是**研究型 change**。`query-rerank-and-persistence` 处理了"输出排名"这个当下卡点（POC：recall@10 0.14→0.81）；本 change 处理更深的"图建得薄/碎"问题。

诊断硬数据（200 篇共享图）：4380 节点 / 2901 边 / 度 1.32、34% 孤立、1681 连通分量（最大 37%）、跨文档边 634/2876。关键认知：**这不是当下检索指标的根因**（reranker 已证 gold 仍进候选集、召回 86%），但它**限制天花板**——真要做需要桥接多篇文档的深多跳，一张碎图走不动。

因此本 change 的纪律是：**先量化、再实验、后定方案**。诊断是唯一硬交付；构建增强（跨文档链接、社区合并、有向/带类型边）都是探索，必须用诊断 + 小规模实验证明增量收益后才落地。

## Goals / Non-Goals

**Goals:**
- 提供可复用的**图质量诊断**（对任意已落盘图：度分布、孤立率、连通分量、跨文档边比例等）
- 用诊断量化"在 reranker 已生效后，图质量改进还能带来多少检索/多跳增量收益"
- 探索并验证 1–2 条性价比最高的构建增强（跨文档链接 / 社区合并）

**Non-Goals:**
- 不做 reranker / 持久化（属 `query-rerank-and-persistence`）
- 不在探索定型前 commit 具体算法或 spec 级构建行为
- 不追 SOTA；有向/带类型边可能拆成更后续 change

## Decisions

### D1: 诊断先行，作为唯一硬交付与回归基线

先做图质量诊断（可对现有 `multihop_bench.db` 直接跑）。任何构建增强都必须用它度量"改善了多少"，避免凭感觉改。

### D2: 构建增强以"假设 → 小规模实验 → 度量"推进，不预先 commit 算法

跨文档链接、社区合并都先当**假设**，在 50–100 篇小图上做对照实验（开/关），用诊断 + 检索指标看增量，再决定是否值得、用哪种实现。

### D3: 复用 CompactionPlugin 协议承载社区合并

若社区合并值得做，接现有 `CompactionPlugin` 协议（对标 fanout_reducer 的 decide_hub，但做社区级），不另造机制。

### D4: 依赖序列化修复

诊断/实验要在已落盘图上跑，依赖 `query-rerank-and-persistence` 的序列化 round-trip 修复（否则 reload 出来的图不可用）。

## Open Questions

- **社区检测算法选型**：轻量连通性/密度启发式 vs Louvain/modularity（需引依赖）vs LLM 辅助（贵）——成本/收益如何权衡？
- **跨文档链接怎么改**：增强阶段② 锚点召回（让新概念更容易连到已有相关节点）？降低 judge_relations 的 create 偏置？还是 build 后加一遍专门的跨文档链接 pass？哪个性价比高？
- **增量收益有多大**：reranker 已把检索指标救起来，图质量改进的**边际收益**到底值不值得投入？必须先用诊断 + 小规模实验量化，可能结论是"暂不值得"。
- **有向/带类型边**：当前边无向无类型；引入方向/关系类型能支撑更深多跳，但成本高——纳入本 change 还是拆出去？
- **孤立节点怎么处理**：34% 孤立节点是该补链接、还是该判定为噪声裁剪（priority_trim 类）？

## Risks / Trade-offs

- **[增量收益可能很小]** reranker 已解决主要痛点 → 图质量改进或许 ROI 低 → 缓解：诊断+小实验先量化，允许"结论是不做"
- **[社区检测引入复杂度/依赖]** Louvain 等需新依赖、调参 → 缓解：先试轻量启发式，不行再升级
- **[改建图影响成本]** 加 pass / 更强锚点召回会增加 build 的 LLM 调用与费用 → 缓解：小规模实验先评估成本，纳入 dry-run 模型
- **[研究型难收口]** 本质不确定 → 缓解：诊断作为可交付的硬产物兜底；构建增强按实验结果增量推进，不强行一次做完
