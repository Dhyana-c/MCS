## Context

`subgraph-bounding` 能力已立"子图 ≤ 上下文容量"的不变量，并要求"递归直到根子集 ≤ 容量"、"LLM 语义归纳（非纯图聚类）"。`seed-graph-directional-hierarchy` 已实现持久虚拟根 + 有向分层 + 抗退化 bounding。

但 100 篇真实建图（`multihop_directed_100_32k`，token_budget=32k）实测：root 直挂 **1284** 个概念、扁平深度 1，**违背了最大上下文子图不变量**。诊断：

- 守门器 `_exceeds_budget` 用 `node.content` 估算邻域（1284×~6≈7700 token < 32k，以为没超），但真实渲染含 name+content（1284×~13≈1.7万）。**估算口径低于渲染口径，系统性低估** → 该触发归纳时没触发。
- `decide_hub`/`_reorganize` 一进一出：一次只造一个 hub，把 1284 个收敛需多轮、撞 `max_reorg` 上限，收敛慢。

## Goals / Non-Goals

**Goals:**
- 把"子图 ≤ 容量"显式化为**最大上下文子图硬不变量**，处处成立（含 root）。
- 守门估算口径 == 渲染口径（含 name==content 去重），杜绝低估。
- 聚类一进多出：一次产出多个主题 hub，邻域一步收敛。
- 渲染 name==content 去重，放宽单窗口容量。

**Non-Goals:**
- 不改语义边双向 / 查询后处理 / 默认 opt-in。
- 不引入向量 embedding 作必需依赖。
- **不引入纯图聚类**替代 LLM 语义归纳（遵循 subgraph-bounding 现有约束）。

## Decisions

### D1：最大上下文子图为硬不变量
任意节点 + 其全部一跳子节点渲染量 ≤ T，**处处成立（含 root）**。守门在邻域逼近 T 时触发裂变，递归直到处处满足。
- **理由**：这是导航 / 查询 / 归纳的共同地基。root 扁平 1284 会让查询从根下钻第一层（`navigate_hub` 渲染全部候选）撑爆窗口。

### D2：估算口径 == 渲染口径（单一渲染/计量函数）
抽出单一"节点渲染"函数，`context_renderer` 与 `_exceeds_budget`/`_select_batch` 共用；估算值 = 实际渲染 token。
- **理由**：口径不一致是 root 违规的根因；**低估比高估更危险**（漏判直接破坏不变量）。
- **备选**：继续两套口径 + 加保守系数——脆弱、治标，弃。

### D3：一进多出聚类
`decide_hub` 输入=中心点+全部一跳子节点（不变量保证装得下），输出多社区；`_reorganize` 一次造多个 hub 分挂。
- **理由**：一进一出收敛慢、撞上限；社区发现天然多簇，应一次落地。
- **难点**：LLM 输出端——逐一回吐上千 id 会撑爆输出且易幻觉（见 D4）。

### D4：输出端控制（主题级 + 紧凑归属）
让 LLM 产出社区划分（主题 + 成员归属），避免逐一回吐海量 id。可选：紧凑编码（成员→社区号）或分批聚 + 按主题标签合并。成员归属须**不重不漏 + 幻觉过滤 + 确定性兜底**。
- **注**：正常态下受不变量约束，一次待聚成员 ≤ 一窗口、规模可控；1284 是违规态，修复 D2 后不再出现。

### D5：聚类必须 LLM 语义，非图聚类
遵循 subgraph-bounding 既有约束：`decide_hub` 语义归纳；图结构仅作辅助信号。
- **理由**：语义边稀疏（实测概念中位度 3）时纯图聚类信号不足，且 spec 明令禁止。

### D6：渲染 name==content 去重
name==content 只写一份（实测大量 extract 概念 name≈content，如 `ConceptDraft(name='Fantasy Football', content='Fantasy Football')`）；估算同口径。
- **理由**：省 token、放宽单窗口容量、使不变量更易维持。

### D7：重叠聚类 + hub 复用，以"降低总量"为统一判据
- **重叠聚类**：去掉"恰好一个社区"的硬划分，一个原始节点 MAY 同属多个 hub，只要重组后总 token / 节点 / 边下降。
- **hub 复用（边吸收）**：某节点的一跳子节点 ⊇ 某 hub 的全部成员时，把它改连该 hub（删到各成员的直接边、加一条到 hub），减边减扇出、复用已有结构。
- **统一判据**：裂变 / 重叠 / 复用等一切重组，净效果 MUST 是降低图总量（token / 节点 / 边）；不降则不落地。
- **理由**：重叠贴合真实（概念常跨主题，如 Elon Musk ∈ 科技 ∩ 政治）；边吸收复用已有 hub 进一步压缩；总量判据给所有重组一个客观成败标准。
- **现状**：均**未实现**——`_reorganize` 当前是"一成员→一 hub"的硬移动，无重叠、无吸收。

### D8：聚类 = 知识重组（合并 / 找关键概念 / 概括）+ hub 对 LLM 同构
- 聚类的本质是对知识**重组**，每个社区按优先级：① **合并同义**——旧的同义概念合并；② **找到关键概念（重点）**——识别社区里的关键概念作组织中心，其余概念关联它；③ **概括成新概念**——无现成关键概念时，把这组概念概括成一个新概念并与旧概念关联。
- 概括出的新概念 MUST 有语义内涵、可独立成义，**禁止空洞聚合标签**（"信息碎片集合""综合信息枢纽"）——这正是万能 hub 的病根。
- 重组后一个原始节点 MAY 关联**多个**父（重叠）。
- `role="hub"` 仅作可观测性标记；渲染给 LLM 时与普通概念**同构**（name+content，无特殊标记）——hub 即"恰好成为组织中心的普通概念"，非特殊类型。
- **理由**：聚类的价值在"找关键概念 / 概括"这种知识重组，而非造分组标签；从源头不产生空洞节点，**取代** `seed-graph-directional-hierarchy` 的 `_is_overbroad_hub` 事后补丁。
- **取舍**："概括出的新概念是否实质"仍需 LLM + 轻校验；一进多出已让杂项分成内聚社区，天然减少"无关键概念、需概括"的情形。

## Risks / Trade-offs

- **[BREAKING 阈值/拓扑变化]** 旧扁平图（`100_32k` 等）与新不变量不一致 → 缓解：文档标注需重建；默认 opt-in 同开关下生效。
- **[LLM 多社区输出可靠性]** 成员漏分 / 幻觉 id → 缓解：解析校验 + 确定性兜底（无法分类者留中心点下，不静默丢）。
- **[口径对齐性能]** 每次估算走真实渲染稍重 → 缓解：单函数 + 必要缓存。
- **[递归震荡]** 聚了又拆 → 缓解：社区清晰度稳定条件 + `max_reorg` 兜底。

## Migration Plan

1. 先做口径统一（D2/D6）——单测断言"估算 == 实际渲染"；默认关时行为不变。
2. `decide_hub` 多社区 schema + `_reorganize` 一进多出（D3/D4）。
3. 100 篇重建验证：root 度从 1735 摊薄、最大深度增加、**每节点一跳邻域 ≤ 容量**。
4. 旧扁平库归档。
- **回滚**：`seed_graph_bounding` 关闭即回退到既有路径。

## Open Questions

- LLM 多社区的输出编码：紧凑成员号 vs 分批合并，哪个更稳？
- name==content 的"相同"判定：严格相等 vs 归一化（trim/lower）后相等？
- 成员无法归类时：留中心点下 vs 单列"杂项"hub（但杂项 hub 又可能逼近 catch-all）？
