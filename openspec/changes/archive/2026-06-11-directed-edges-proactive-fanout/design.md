## Context

当前 MCS 用**两套边类型**：层级边 `out`（父→子 + 成员上行）与语义边 `bidirectional`。守门（`fanout_reducer`）在邻域**已经超 T 后**才被动触发，超量时被迫**分批**（≤50）喂 `decide_hub`；`get_neighbors` 对 `out` 有向、对 `bidirectional` 无向，`_reorganize` 用 `delete_edge` 重连会把双向语义边**退化**成单向。

全量建图实测（见 proposal）：`__seed_root__` 退化成扁平直挂 ~800 子节点、贴着 T 不收敛；裂变 118 次、`token 总量未下降` 118 次、0 回滚；decide_hub 平均覆盖 48%。本设计从边模型与守门时机两处根治。

依据宪法 `T ≈ W/2`（T 为 bounding 预算，W 为上下文窗口）：只要邻域始终 ≤ T，则"中心 + 全部一跳子节点"必能塞进完整窗口 W 并留出 LLM 推理/输出余量——这是"整窗单次、不分批"成立的前提。

## Goals / Non-Goals

**Goals:**
- 取消双向边：全图单向，语义关系=两条对向单向边；存储 schema 简化为 `(source_id, target_id)`。
- 守门改主动：邻域**即将超 T 即触发**，始终 ≤ T；裂变**整窗单次、不分批**；**全局**任意节点适用。
- 裂变动作由**单次整窗 LLM 生成**产出 hub（新建/提拔）+ 已有节点关联 + 合并同义；多父、幻觉忽略、未选留存、token 降才接受。
- 修复 `_validate_reorg` before/after 同口径，让"总量降才落地"真正生效。
- 语义边参与导航（提升可达性）。

**Non-Goals:**
- 不改摄入前 5 段（extract/judge/apply 的概念抽取与决策逻辑保持）；新概念仍无条件挂 root（用户确认可接受）。
- 不在本次引入"按相关 hub 直接归位"的摄入改造（另议）。
- 不改 token 估算器实现本身（铁律一口径一致性保持）。

## Decisions

### D1：取消双向边，语义关系=两条对向单向边
**选择**：删除 `direction="bidirectional"`；`judge_relations` 的每条语义关系落 `a→b` 与 `b→a` 两条单向边。
**理由**：双向边带来"无向邻接 vs 有向邻接"两套语义、`_edge_key` 端点归一化、裂变退化等复杂度；语义关系 `a-b` 在语义上等价于 `{a→b, b→a}`，拆两条后全图只剩单向边、原语唯一。
**备选**：保留双向边、仅修裂变不退化它——否决：仍留两套遍历语义、且"语义边参与导航"难统一表达。

### D2：存储 schema 收敛为 `(source_id, target_id)`
**选择**：边表去掉 `direction` 列；`_edge_key=(source,target)`；`add_edge(source,target)` 仅维护 `adjacency[source].add(target)`；`get_neighbors==get_out_neighbors`。
**理由**：单向后 `direction` 维度无意义；邻接退化为单一有向表，CRUD 与 round-trip 都更简单。
**备选**：保留 `direction` 列恒为 `"out"`——否决：冗余、误导后人以为还有别的方向。
**迁移**：旧库含 `bidirectional` 行，不迁移、直接重建（见 Migration）。

### D3：层级骨架由节点 `role` 识别，取消成员上行边
**选择**：层级仅保留下行 `父→子`；不再建 `成员→原父` 上行边。裂变/导航/hub 复用判定"谁是 hub"改读 `node.role=="hub"`，不再靠边类型。
**理由**：单向无类型后，边本身不携带"层级 vs 语义"信息；`role` 已存在且天然标记组织中心（宪法："hub 是恰好成为组织中心的普通概念"）。上行边的唯一作用（记录原父）在纯下行导航下不再需要，语义关系由独立的两条语义边承载。
**备选**：给边加 `kind` 列（hierarchical/semantic）——否决：与"越简单越好、起点终点足矣"的目标相悖；role 已够用。

### D4：守门主动触发 + 整窗单次、不分批
**选择**：阶段 ⑤.5/⑥ 在某节点一跳邻域**即将超 T** 时即触发裂变；触发时把"中心 + 全部一跳子节点"一次性喂 `decide_hub`（必 ≤ W）；删除 `_select_batch` 贪婪取批与 `_decide_hub` 折半重试。任意节点（不限 root）超预算都按此处理。
**理由**：被动滞后 + 分批是 stub 扁平堆积、48% 覆盖、慢漏的直接成因；主动保持 ≤ T 使全部子节点恒能一次装下，LLM 一次看到全局、决策更稳更全。
**备选**：保留分批但修覆盖率——否决：分批本身让 LLM 看不到全局、且物理上只能在已超 T 后补救。

### D5：裂变动作 = 单次整窗 LLM 生成（hub + 关联 + 合并）
**选择**：一次 `decide_hub` 调用，LLM 在整窗里同时给出：若干 hub（新合成 / 提拔已有子节点）、每个 hub 关联哪些已有节点、哪些已有节点合并同义。重连：对每条有效 `(hub H, 成员 M)` 做 删 `C→M`、建 `C→H`、建 `H→M`。
**规则**：被关联节点不存在（幻觉 id）→ 忽略该条；LLM 未关联的子节点 → 留在 C 下不丢；一个已有节点可关联多个 hub（多父/重叠）；合并同义在同一次生成里产出、非独立第二遍。
**理由**：保留宪法铁律二三策略（合并/提拔/概括），且"合并基于整窗全局判断"质量更高。
**备选**：合并单独再跑一轮——否决：用户明确"基于整个生成"，且多轮增加 LLM 调用与不一致风险。

### D6：接受判据 before/after 同口径
**选择**：`_validate_reorg` 的 `before` 与 `after` **都按中心节点 C 的全邻域**用同一估算器计算；仅当 `after < before` 才落地，否则回滚。
**理由**：现状 before 只算一批（~1301）、after 算全邻域（~31590），结构性恒判失败、护栏失效。同口径后"总量降才落地"（宪法优化判据）才真正生效。

### D7：导航/扩展沿全部单向出边
**选择**：`hub_fallback` 下钻与 `query_engine._traverse` 扩展都走单向出边；语义边（两条对向单向）因此参与导航。防环仍靠 `visited` + `max_depth`。
**理由**：单向后无"层级 vs 语义"边类型可据以限制；语义边参与导航可提升种子可达性。
**风险**：见下（导航漂移）。

## Risks / Trade-offs

- **导航沿语义边漂移** → 种子定位可能从层级骨架漂到语义旁系、降低精度。缓解：`visited`+`max_depth` 封顶；**必须用多跳评测对比改造前后召回**，不达标则在 spec 增"语义边导航深度/优先级"约束（列入 Open Questions）。
- **stub 节点 + 纯 token 触发仍可能较宽** → 节点很小且单次 LLM 欠挑时，C 邻域刚回落 ≤ T 即停、仍可能宽。缓解：D6 让"token 降即接受"可增量收敛；必要时引入**扇出数硬上限 N** 作协同判据（Open Questions）。
- **存储/边数增加** → 语义边翻倍（每关系 2 行）。可接受：语义边稀疏，且换来原语统一。
- **导航上行通路减少** → 取消上行边后，从子节点回祖先的路径仅靠语义边。可接受：导航本就自顶向下。
- **大批量整窗喂 LLM 的稳定性** → 即便 ≤ W，几百个子节点一次喂仍可能压力大。缓解：主动触发使单节点子树规模受控（远小于历史 800）；评测观察 `decide_hub` 空响应率。

## Migration Plan

1. 改 store/edge 模型与 fanout 后，旧 `*.db`（含 `bidirectional` 行、上行边、退化扁平 root）**作废**。
2. 重跑建图脚本从零重建 `bench/multihop_rag/outputs/...`（脚本无需改，仅删旧库重跑）。
3. 回归：`.venv\Scripts\python.exe -m pytest -q` 须绿（含 store/fanout/seed-graph/导航相关用例，可能需同步更新断言）。
4. 评测：多跳评测对比改造前后召回/精度，确认"语义边参与导航"非负向。
5. 回滚策略：本次为 spec-driven change，未归档前可整体弃用 change、还原代码；旧库已无价值不需回滚数据。

## Open Questions

- 是否需要在"即将超 T"之外，再加一条**扇出数硬上限 N**（N 取"N 个全文节点渲染 ≤ T"）作为协同触发判据，以彻底杜绝 stub 扁平？倾向：先只用 token 触发跑评测，按数据决定。
- 语义边参与导航若引起漂移，是否在 `navigate_hub` 候选里**区分层级出边与语义出边的优先级**（虽不分边类型存储，但可按对端 `role` 或是否成对存在 `b→a` 推断）？留待评测后定。
- 合并同义时，被合并节点的**两条语义边**如何迁移到代表节点而不产生重复/自环——实现细节，在 tasks 里覆盖。
