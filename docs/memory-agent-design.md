# MCS 记忆 Agent 设计文档（v0.2 草案）

> 本文把「记忆 Agent 与置信度算法规则」落成可评审的设计。
> **v0.1 → v0.2 核心变化**：数据模型从「概念层 + Fact 独立表 + 事件表」重构为
> **`concept / fact / event` 三类节点 + 超边式事实**。事实不再是带 label 的边
> （`property_graph` 模式退役），而是**居中连多概念的无向节点**（超边节点化）。
> 本版按 **「单独分支迭代、暂不考虑向后兼容」** 前提重写。
> 标注约定：✅ 现状已有可复用 ｜ 🔧 现状需改 ｜ ➕ 需新增 ｜ ⚠️ 待验证 ｜ ❓ 待决策
> 参考规范：`openspec/specs/`（`subgraph-bounding`、`architecture`、`query-pipeline`、`write-pipeline`）。

---

## 0. 定位与前提

**定位**：独立记忆 agent，自有 agent loop，仅暴露记忆类工具（学习 / 查询 / 推理 / 联想），基于 MCS 图底座做通用问答。

**差异化（唯一，所有规则为它服务）**：任意概念的一跳邻域按构造 ≤ T，检索 primitive 无需剪枝即返回 context-bounded 结果。凡是会让概念邻域膨胀、破坏这条不变量的设计，一律否决。

> **前提变更**：v0.1 以「在现状代码上最小接缝」为纲；v0.2 转为 **新分支重做**（事实表示根本改变）。故下文「现状映射」性质从「接缝点」变为「重做对照面」——标注现状代码位置，是为了说明重做哪些既有模块，而非贴补。

### 0.1 命门（来自 v0.1 §7，必须前置，仍成立）

差异化前提 **「bounded 无剪枝 ≤ 无界 + 剪枝（同准确率更省 token/turn）」未验证**。且存在**已知反例**：事实侧有界最终要靠 priority 截断兜底（见 §2），截断会让低活跃邻居（如 10 万员工中的冷门项）被砍、点查不到——bounded 省 token 与 bounded 丢召回是同一枚硬币的两面。

> **结论**：C0 阶段必须用对比实验**同时量召回与 token**，且必须包含「需要冷门邻居」的查询子集。未验证前，§1–§6 为图纸，§9 的 C1–C5 不建。

### 0.2 本版新增的关键认知

- **事实变节点后，概念邻域里最重、最容易超 T 的正是事实节点本身**（每个事实节点带 claim + 扩展，比带 label 的 fact 边重）。这放大了 §0.1 命门。
- 本版对此的回应不是「聚类事实」（聚类复杂、丢信息、坏归属——见 §2.1），而是 **保真不聚类 + 查询期 hub 折叠 + 截断兜底**（§2）。这是本版相对 v0.1 的核心机制创新，但折叠的覆盖范围有限（§2.3），命门仍未真正关闭——仍需 C0 验证。

## 1. 数据模型：三类节点 + 超边式事实

三类节点：**concept / fact / event**。hub 不作为独立 `role`（见 §1.5）。概念之间**无直接语义边**——任何语义关系都经 fact 节点居中表达。`property_graph`（带 label 的 `kind="fact"` 边）退役。

### 1.1 概念层 Concept（可导航、薄、bounded）

即 MCS 现有概念层，检索 / 联想主结构。`role="concept"`，用户实体也是 concept（MCS 不区分概念与实体）。

| 规则 | 状态 | 说明 |
|---|---|---|
| C1 概念节点保持薄（name + content），置信不直接挂节点 | ✅ 现状即如此（`graph.py:14-26`） | 置信落 fact 节点、不落 concept |
| C2 概念之间无直接语义边，关系必经 fact 节点 | 🔧 重做 | 现 `property_graph` 的 `kind="fact"` 概念-概念边废弃 |

### 1.2 事实层 Fact（超边节点，居中、无向、保真不聚类）

**事实 = 超边的节点化**（"既是边又是点"）：一个 fact 节点连接 N 个 concept（N 元关系，无向、允许无限端点），承载这条断言的全部语义与属性。事件直接连到 fact 节点（绕开"边连边"）。

```
Fact {
  id
  claim                     # 规范化命题（节点 content），承载关系语义与方向（见 §1.4）
  confidence                # 派生值（§3），落节点扩展
  valid_interval            # bi-temporal 有效期，落节点扩展
  origin                    # 继承自主导背书事件，落节点扩展
  # supporting_events 不在节点字段里——由 event→fact 背书边表达（§1.4）
}
```

| 规则 | 状态 | 说明 |
|---|---|---|
| F1 事实是节点、居中连多 concept、无向、无限端点 | ➕ 新节点类型 `role="fact"` + 新连接边 | concept↔fact 无向；超边星型 |
| F2 冲突共存不互删、不聚类、裁决只改 confidence 排序 | ➕ 多版本 = 多个 fact 节点各自连同样端点 | 保真：原始事实永不删；玻璃箱，败方保留低置信 |
| F3 事实不作为查询焦点 | ➕ 约定 | fact 节点只在 concept 视图里被渲染，不作为遍历起点 / 焦点；其自身视图不定义有界（§2.4） |
| F4 事实永不聚类 | ✅ 沿用铁律 | 归属 / 置信 / 有效期精确到每条事实；减量靠 §2 的 hub 折叠，不动事实本身 |

> **为什么事实是节点而不是带 label 的边**：① 可挂置信度 / 有效期 / 背书（边只有 label + priority，塞不下）；② N 元关系天然（一节点连 N 端）；③ 多版本冲突 = 多节点（F2）；④ 事件背书直接连节点，**精确锚定到具体版本**（v0.1 的 D1 缝隙由此消失）；⑤ 边退化为纯连接，语义统一在 fact 节点 claim 里。

### 1.3 事件层 Event（进图节点，极简，不进概念邻域）

事件 = 原始输入。极简四要素，其余关联一律用边表达，不堆字段。

```
Event {
  type              # 预留：输入形态（对话 / 文件 / feed / …），非 origin
  time              # 接收时间（= ingested_at；对话记忆场景）
  raw               # 原始输入（可能很大，见 §2.5 入库策略）
  user              # → 用户概念节点（用户 = concept 实体）
}
```

| 规则 | 状态 | 说明 |
|---|---|---|
| E1 只追加永不改 / 删 | ➕ 事件节点 append-only | "更正" = 新事件 + 旧事件标 `invalid-after` |
| E2 事件不进概念活跃邻域、不参与 T | ➕ 事件节点不被 concept 的 getter 触达 | `_node_view` / `estimate_active_view` 取不到 event |
| E3 事件的"输入用户"指向 concept 实体 | ➕ event→user 关联 | 用户信誉挂用户节点扩展，参与置信度（§3.1） |

> **断言时间口径**：事件层只存 `time = 接收时间`；`valid_time`（内容里提到的时间）下沉到 fact 节点的 `valid_interval`，随事实抽取。这让 MCS 定位收敛为"个人对话记忆 agent"。

### 1.4 边模型

全图三类边：

| 边 | 方向 | label | 语义 | 状态 |
|---|---|---|---|---|
| `hierarchy` | 父 concept → 子 concept（单向） | 无 | 层级骨架，驱动导航 / hub / fanout。**唯一的概念-概念直接边**（结构性，非语义关系） | ✅ 沿用 |
| concept ↔ fact | **无向** | 无 | 事实连接。fact 居中连 N 个 concept；超边星型 | ➕ 新 |
| event → fact | 单向（背书） | 无 | 事件背书事实。**fact 侧可反查**（置信计算汇总背书事件）；**event 不进任何 concept 视图** | ➕ 新 |

> **方向问题（❓ 待决策 / 已知限制）**：concept↔fact 无向，丢了主宾（"A 雇佣 B" ≠ "B 雇佣 A"）。当前接法：**方向靠 fact 节点 claim 文本表达**（content 里写清主宾）。若后续需要结构化方向，给 concept↔fact 边加 `role` 扩展（subject / object）。对话记忆场景倾向先用 claim 文本，结构化方向列为待决。
>
> **无向边存储（❓ 待决策）**：MCS 现状全图边单向 `source→target`。无向边新分支需定实现：双向存储 or 单向存储 + 双向查询。倾向后者（存储单向、查询双向，省一份）。
>
> **背书锚定**：event→fact 边直接连到 fact 节点 = 精确到具体事实版本（v0.1 D1 缝隙解决）。

### 1.5 hub（概念层涌现，事实不参与）

- `role ∈ {concept, fact, event}`，**hub 不占 role**。
- hub 是**概念节点**的涌现标记：概念出边侧（层级子节点）超 T → `decide_hub` 把一组概念聚成社区 → 重组出 hub 当组织中心。hub 生存在于 `hierarchy` 层级骨架里。
- **fact 节点不作 hub**：fact 属关系侧、不进层级、不参与 fanout（同 `attribute_node` 模式的属性节点处理）。让 fact 当 hub 会把关系侧拉进层级侧，混淆两侧。
- hub 标记仅供系统识别层级 / 可观测 / 复用逻辑；渲染给 LLM 时与普通 concept 无异（CLAUDE.md 已如此，不变）。
- "看大模型自己决定"不变：`decide_hub` 仍是 LLM 语义归纳，候选池是 concept。

> **hub 机制照搬现状**：`decide_hub` 裂变、hub 复用（边吸收）等不变。新模型只把"语义关系"从 hub / fanout 里剥离干净——fanout 只动 `hierarchy`、不碰 fact 节点。

## 2. 有界机制：hub 折叠 + 截断兜底

事实侧怎么在 ≤ T 内呈现？本版的核心回答。

### 2.1 不聚类事实（铁律沿用）

CLAUDE.md 铁律"入边不聚类，会坏归属语义"——本版**逐字沿用**：fact 节点永不聚类。理由：聚类会抹平每条事实的背书事件 / 置信度 / 有效期，而裁决（§3.5）和置信计算依赖精确到事实。聚类也复杂、丢信息。

### 2.2 减量靠查询期 hub 折叠

事实保真存储，但**渲染时借用 hub 的归纳成果**：

> **折叠规则**：查询期渲染一个 hub H（或其成员）的邻居视图时，对每条经 fact 节点的连接，**若该 fact 的两端 concept 都在 H 的成员集内，则在该视图下跳过（折叠）**——它们已被 H 组织，关系隐含在"同属 H"，不重复逐条展开。

- 折叠**只在渲染 / 估算时生效，存储保真**；下钻可取全部事实。是"省略"不是"删除"。
- 这把 hub 的价值二次利用到事实渲染：hub 不仅组织了概念层级，还顺手"吸收"了成员间的事实细节。
- 效力**依赖于 hub 化质量**：hub 化越好，越多 fact 的两端落在同一社区，越多被折叠。

### 2.3 截断作最后兜底（⚠️ 命门未关）

折叠**只覆盖"两端同属一个 hub 成员集"的事实**。两类情况折叠帮不上：

1. 一个 concept 对大量**跨 hub 的独立对象**有事实；
2. 这些对象**未被 hub 化**。

此时 concept 视图仍超 T，兜底只能是 **priority 截断**（按 fact 的 confidence / priority 砍低优先级，砍了就丢）。

> **这是 §0.1 命门的延续**：折叠为主、截断兜底，双层。截断必然丢冷门——记忆 agent 又要"努力够冷门"（§3.3 P3）。**截断到底丢多少、可不可接受，仍要 C0 实验验证。** 本版没有真正关闭命门，只是把"何时丢"从"事实多就丢"收窄到"跨 hub / 未 hub 化才丢"。

### 2.4 折叠语义（建议，❓ 待敲定）

- 折叠只作用于**概览视图**；**高 priority 事实不折叠**（避免误伤成员间的高价值非冗余事实）。
- 折叠 ≠ 删除：下钻始终可取被折叠的事实。
- fact 节点不作查询焦点（F3），故不需为"fact 自身视图"定义有界——它只在 concept 视图里作为元素被渲染 / 折叠。

### 2.5 铁律一：折叠必须同步进估算（🔧 实现核心）

CLAUDE.md 铁律一"估算口径 == 渲染口径"。新增硬约束：

> **估算 concept 活跃视图 token 时，必须先应用与渲染完全相同的折叠规则，再计 token。** 漏折叠 = 低估 = 不变量失守（1284 坑的同类）。

fact 节点在 concept 视图里按什么计 token（邻居节点：claim + 扩展）需与渲染逐字一致。折叠规则是估算与渲染的**共用前置**。

### 2.6 事件 raw 的入库（❓ 待决策）

event 的 `raw` 可能是整个文件、很大。**不应进 event 节点 content**（一旦被任何渲染 / 估算路径触达，`estimate` 直接爆）。建议：raw 存事件表 blob 或外部引用，event 节点只存指针 + 四要素。与 E2（事件不进概念邻域）同源，但 raw 入库策略要单独定。

## 3. Origin 分流（反投毒第一道结构）

origin 落在**事件**（事件的 type / 来源域）与**用户**（用户 concept 节点的信誉），经 event→fact 背书链流入 fact 置信。

| origin | 来源 | 置信处理 |
|---|---|---|
| `ingested` | 外部源（书 / 站 / feed） | 来源信誉加权印证 + LLM 伪计数先验（§3.3） |
| `derived` | 从高置信图内事实演绎 | 置信从前提流入，不另设来源信誉、不施 LLM 先验 |
| `recalled` | agent 参数 / 世界知识，未落图 | 当作单一低信誉来源；无自评先验；须外部印证才能高置信 |

| 规则 | 状态 | 说明 |
|---|---|---|
| O1 agent 是来源但不享全局高信誉 | ➕ 来源分级 | 复用事件 type / 用户 concept 作 source 雏形 |
| O2 涉及自身 `recalled` 事实的冲突裁决，agent 不作唯一 / 高权重裁判 | ➕ 裁决器票权 = 其低信誉 | §3.5 |
| O3 `recalled` 不施 LLM 自评先验 | ➕ 先验仅施 `ingested` | §3.3 P1 |

## 4. 置信度算法（truth-discovery / 数据融合家族）

**核心戒律：置信度永不来自单次推理行为自身；只来自跨独立来源印证、来源信誉、时效。** confidence 落 **fact 节点扩展**（非 concept、非 edge.priority）。来源信誉落**用户 concept 节点扩展**。

### 4.1 来源信誉 r(s)

`r(s) ∈ [0,1]`，域级 / feed 级（不做页级，太稀疏）。挂在用户 concept 节点扩展（如 `extensions["reputation"]`）。可外部先验 bootstrap。

### 4.2 依赖折叠（强制）

印证前必须把背书事件（经 event→fact 边汇总）按独立性聚类，塌掉副本：

```
groups = dependence_cluster(supporting_events)   # 转述/引用同一上游 → 同组
W_plus(f) = Σ over groups g: rep(g)              # 每组只取代表信誉，组内不累加
```

> **规则 C-FOLD**：无依赖折叠的印证 = 流行度放大器（谁被抄多谁赢），与真值相反。任何多源场景此步承重。
>
> **❓ `rep(g)` 语义待定**：组内最大信誉？代表源信誉？均值？——比"参数标定"更基础，需先定义。

### 4.3 LLM 似然先验（伪计数，有限、有条件）

```
prior_weight(f) = κ · domain_scale(f) · p_llm(f)
# κ = 等效样本量（伪计数），证据一多即被淹没
# domain_scale ∈ [0,1] = 是否在模型能力分布内；专有/小众域 → 0
```

| 规则 | 说明 |
|---|---|
| P1 | 仅施 `origin=ingested`（`derived` 用前提，`recalled` 不施，见 O3） |
| P2 | 用伪计数非固定比例——弱证据先验主导，强证据先验退场 |
| P3 | `p_llm` ≈ 训练频率，会压制冷门真相、抬升流行谬误，与"努力够冷门"目标相反 → **必须可被证据淹没且按域缩放** |

### 4.4 后验置信（起始形式，参数待标定）

```
conf(f, t) = ( W_plus(f) + prior_weight(f) )
           / ( W_plus(f) + W_minus(f) + κ·domain_scale(f) + ε )
```

性质：单源时先验主导；印证累积时先验淡出；冲突拉低置信。结果落 **fact 节点扩展 `confidence`**。

### 4.5 裁决与信誉更新

```
best = argmax_i conf(f_i, t)        # 当前最优；败方不删，保留低置信（多 fact 节点共存）
# 仅当 (margin 足够大) 且 (分歧不能被时效解释) 时，才更新败方来源信誉：
r(s) ← r(s) + η · (outcome - r(s))  # η 小、平滑、有界
```

| 规则 | 说明 |
|---|---|
| J1 时间感知 | 来源"当时为真、现已过期"不算撒谎，不降信誉；只惩罚"当时即错" |
| J2 防自激 | η 小步 + 先验平滑，避免"被压低→更易输→更被压低"的马太螺旋 |
| J3 | 裁决只在 margin 超阈值时触发信誉更新；否则标 `uncertain-coexist`，不强分胜负、不动信誉 |

### 4.6 Bi-temporal

每 fact 节点带 `valid_interval`。置信与裁决按查询参考时间 t 求值。被取代的 fact 不删，标 `invalid-after`。

> **confidence 不再落 edge.priority**：v0.1 曾想 confidence → `edge.priority` seam。v0.2 事实变节点，confidence 直接是 fact 节点扩展字段，不再借用 edge.priority。`edge.priority` 的语义（Phase 2 由 `PriorityScorer` 从边扩展算）不受影响——concept↔fact 连接边若需要 priority（用于 §2.3 截断排序），由 scorer 从该边扩展算，与 fact 节点的 confidence 是两个粒度。

## 5. 检索、工具与努力程度

### 5.1 工具集

| 工具 | 语义 | 状态 |
|---|---|---|
| `query(text)` | 概念入口 + 子图扩展（事实经 fact 节点两跳：concept→fact→concept） | 🔧 改 `_node_view` 两跳 + 折叠渲染 |
| `associate(node)` | 一跳邻域 bounded（含折叠后的事实） | 🔧 改 `_node_view`（`query_engine.py:286`） |
| `search(text)` | 别名 / 关键词倒排 = grep | ✅ `alias_index` + `EntryPlugin`（`config.py:27-32`） |
| `reason(a,b)` | 路径 / 桥接 | ⏸ 默认不实现 |
| 写入工具 | 独立成路（§6），**不在 QA loop 内** | 🔧 `WritePipeline.ingest` |

> **fact 不作焦点（F3）**：工具不暴露"以 fact 为起点"的遍历；fact 只在 concept 视图里被看到。

### 5.2 努力程度 = 一根合成轴

努力 `e` 同时控制：① 置信地板 θ(e)（单调递减：低努力→高地板；最大努力→地板降到底，纳入争议 / 冷门低置信）；② loop 预算；③ 遍历深度 / 广度；④ 工作 context 大小。

| 规则 | 说明 | 状态 |
|---|---|---|
| EF1 排序不切断 | 置信用于**排序**；θ(e) 决定纳入深度。禁止硬阈值二元开关 | 🔧 截断作用于 **fact 获取层**（concept 取其 fact 邻居时按 confidence + θ(e)），**不是**种子节点的 `PriorityTrim`（v0.1 接错层） |
| EF2 永不静默省略 | 按 θ(e) 过滤后空 / 过薄，不得返回空；返回最高可得置信项 + 标注 | ➕ 查询期行为 |
| EF3 不动 T | 努力放大工作 context，**绝不改结构不变量 T** | ➕ 查询期独立预算（写读共用同一 `TokenBudget`，`builder.py:77→:118/:129`，effort 另建第二预算） |

> **EF3 代码根因**：写入期守门（`decide_hub` 判 T）与查询期渲染共用同一 T。改它 = 已存图（按 T 归纳）与新写入（按新 T 裂变）混 T，性质同混库。effort 调"查询窗口"必须走第二预算。

## 6. 写入管线（廉价在前，昂贵在后）

## 7. 反投毒闭环（汇总）

1. 读写分离：QA 只读，写入独立 gate（W2）。
2. agent 是来源、不享高信誉、产出按 origin 分流（O1–O3）。
3. `recalled` 须外部印证才能高置信；agent 不当自己的高权重裁判。
4. 置信只来自印证 / 信誉 / 时效，不来自单次推理（§4 戒律）。
5. 冲突共存、不互删（多个 fact 节点）；裁决只改排序（F2）。

## 10. 风险与待决项（诚实标注，**不要当已成立**）

### 10.1 已和用户对齐（本版定型）
- ✅ 三类节点 `concept / fact / event`；概念间无直接语义边；`property_graph` 退役。
- ✅ 事实 = 超边节点（居中、无向、无限端点、保真不聚类）；事件直接连 fact 节点（背书锚定版本）。
- ✅ 事件极简 `{type, time, raw, user→用户 concept}`；`valid_time` 下沉 fact 节点。
- ✅ hub 不占 role、只在概念层涌现、fact 不作 hub；hub 机制照搬现状。
- ✅ 减量 = 查询期 hub 折叠 + 截断兜底；事实不聚类（铁律沿用）；fact 不作查询焦点。
- ✅ confidence 落 fact 节点扩展，不落 edge.priority；来源信誉落用户 concept 节点。
- ✅ 单独分支迭代、暂不考虑向后兼容。
- ✅ effort 不动 T（写读共用实例根因）。

### 10.2 待验证（命门）
- ⚠️ **核心实验未跑**：bounded + hub 折叠 是否在同准确率（含冷门召回）下省 token/turn 打赢"无界+剪枝"？尤其**折叠只覆盖同 hub 内事实，跨 hub / 未 hub 化仍靠截断丢冷门**——这部分召回损失多大，未量化。**未验证前其余为图纸（§0.1）。**

### 10.3 待决策（影响骨架，需用户拍板）
- ❓ **D1 concept↔fact 方向**：无向丢主宾，靠 claim 文本够吗？还是要连接边加 `role`（subject/object）结构化？
- ❓ **D2 无向边存储**：双向存储 or 单向 + 双向查询？
- ❓ **D3 折叠语义**：高 priority 事实是否豁免折叠？折叠粒度（概览 vs 全量）？
- ❓ **D4 事件 raw 入库**：节点 content / 事件表 blob / 外部引用？
- ❓ **D5 置信反馈回路**：查询→活跃度 / 信誉更新改变"查询只读"假设，走独立通道——可接受吗？
- ❓ **D6 rep(g) 语义**（§4.2）：依赖折叠的组代表信誉怎么取？

### 10.4 承重但未设计
- 矛盾检测 / 事实对齐：皆 LLM 判断、会错。兜底：LLM 不确定 → 标 `uncertain-coexist`，不自动裁决、不动信誉（J3）。
- 参数标定：κ、η、θ(e)、domain_scale、aggregate 选型，均需经验数据。
- 写侧成本：每条入库跑 LLM 段；多源下量级可控，仍需实测。

### 10.5 难域（留到底座证明后）
