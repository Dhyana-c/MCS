# 最大上下文子图（Maximum-Context Subgraph, MCS）技术方案

> 一种面向单一领域、由大模型语义驱动的知识图谱与检索机制。不依赖 embedding / 向量检索，靠大模型直接阅读"装得下的局部子图"来完成关系发现、聚类与查询。

---

## 0. 设计目标与赌注

MCS 要解决的问题：把一个领域的知识组织成一张图，让大模型能在**永远不爆上下文窗口**的前提下，对其中的知识做检索和推理。

它押的一个赌注是：**知识有足够的局部性**——回答一个问题所需要的概念，在图里也彼此靠近，靠几跳语义游走就能连到一起。这个赌注对"已经能被人类整理成可教结构的领域"（物理、工程、各类有教科书/本体的学科）最成立；对跨多个领域综合、或强语境依赖、矛盾常态化的知识（法律、历史、文化、未定型的前沿）会发紧。

一个贯穿全文的原则：**大模型是唯一的语义引擎**。所有"这俩概念相关吗""谁更一般""该往哪走"的判断，都由大模型阅读真实内容做出，不用向量相似度兜底——因为向量在实际中不可靠，而这套机制的全部意义就是用"读真东西"取代"算相似度"。

---

## 1. 整体结构：两层

**事实层（Event Layer）**：原始事实的线性时序日志，只追加、不修改。每条事实**有向地**锚定它提到的若干概念（事实 → 概念）。它是知识的来源，也可作为"最近聊过什么"的查询热启动入口。

**概念层（Concept Layer）**：从事实中提炼出来的语义网络，是 MCS 的主体。它是一张**一般有向图，允许环**——不是树，也不是 DAG。下面所有的不变式、聚类、查询都作用在这一层。

两层的关系：事实是情景记忆（episodic），概念是被蒸馏出来的语义层（semantic），二者之间的连线天然是有向的（事实引用概念）。

---

## 2. 基本定义

### 2.1 节点

- **概念节点**：一个概念。带有：
  - `content`：该概念在**当前抽象层**上的完整描述；
  - `summary`：紧凑摘要，用于被别处列举/累积时的低成本表示；
  - `name` + `aliases`：规范名称与别名集合。别名应在概念创建时由大模型**主动生成**（同义词、缩写、常见说法、易错写法）。别名不是小字段——它决定了查询时有多大比例能走便宜的词法入口、多大比例掉进昂贵的顶层兜底，本质上**别名集合就是你这套词法索引的质量本身**。
  - 概念分两类角色：**实体概念**（小明、红色）与**属性/关系节点**（小明的爱好）。属性节点持有一个**版本列表**（见 §2.2、§4.6）。
- **属性/关系节点**：把"小明的爱好"这类关系具体化成的节点，上面挂着带 `{时间, 置信, 出处, 是否 superseded}` 的多条**版本/说法**（小明的爱好：红@t1、蓝@t2）。版本列表受容量约束，随版本化与淘汰维护（§4.6）。
- **事实节点**：一条原始事实，属于事实层，只追加。它锚定（有向指向）它提到的若干概念节点，并带时间戳，供时序召回使用。

### 2.2 边、属性节点与版本

> **实现标注（`relation_model`）**：本节描述的"无类型邻接边 + 属性节点"模型即 MCS 的
> **`attribute_node` 模式**（Phase 1 选项，默认关闭）。它与默认的 **`property_graph` 模式**
> （带 label 事实边 `主 —谓→ 宾`）双模式并存，由 `MCSConfig.relation_model` 在建图时选定。
> 本节所述**版本 / superseded / 出处 / 置信**属 Phase 2；Phase 1 的 `attribute_node` 模式
> 属性节点仅持**单一当前说法**、不带版本。详见 `openspec/changes/attribute-node-model`。

**只有一种边：无类型的邻接边。** 不预设方向、**初始双向**；**方向/非对称靠合并涌现**——一片区域塌缩成以枢纽为中心的星型时，枢纽 → 成员的方向才定下来（见 §4.4）。边只表达"这两个概念相关 / 被一同提到"，不表达"怎么相关"。

**关系/属性被具体化成自己的节点，版本挂在它上面。**"小明喜欢红色"不是在小明上挂个标签，而是立一个**属性节点**「小明的爱好」，小明、红色都连到它（无类型边）；这条关系的**各个版本**——红@t1、蓝@t2，每条带 `{时间, 置信, 出处, 是否 superseded}`——挂在「小明的爱好」上。好处：① **反向查询**从红色经「小明的爱好」回到小明，不必读小明的全部邻居；② 小明节点不被一堆版本撑大；③「小明的爱好」本身是概念节点，要不要跟已有属性节点合并/归一，直接复用概念去重（§4.5），不用单独搞谓词归一。

**粒度是个旋钮**：粗（「小明的爱好」，节点少，但会混挂红/蓝/摇滚等不同侧面，靠**查询时仲裁**分辨"同槽互斥"与"异侧共存"，见 §5.4）vs 细（「小明喜欢的颜色」，节点多、仲裁少）。

> **为什么不给边加谓词。** 谓词把每条断言逼进 (主−谓−宾) 三元组，要抽谓词、迟早要归一化（喜欢/偏爱/钟意），又脆又是不想要的判断；具体化成属性节点 + 自然语言"版本/说法"更自然：模型直接读、不用抽不用归一、能装下三元组装不下的复杂陈述，版本化也天然。它**仍然满足"关系要有含义"**——含义在属性节点的版本里，边回归纯邻接。

### 2.3 枢纽（中转节点）是"角色"，不是"物种"

当一个节点的邻居多到一跳装不下时，需要引入中间一层来降低扇出。这一层可以是：

- **被提拔的现有概念**（优先）：如果这批邻居里本就存在一个能收纳其余几个的上位概念，就把它提上来当中介。它是真概念、带完整内容。
- **合成枢纽**（兜底）：如果这批邻居确实平级、没有现成上位概念，才临时造一个匿名节点占位，它只带一段摘要。合成枢纽是**临时的、有损的占位**——等真正对应的概念哪天被插进来，它就该被取代/吸收。

因此"内容节点 vs 枢纽节点"不是两种东西，而是一个节点**此刻是否在扮演中介**。

### 2.4 Token 预算 T

设领域上下文窗口的可用长度为 W，则 **T ≈ W / 2**（实际再留一点给指令和输出，略小于一半）。

取一半的理由有两个：
1. **合并**两块子图时要同时把两块读进来比较，各占一半正好都装下。
2. **查询**时只读一块（≤ 半窗口），剩下一半留给大模型推理，一鱼两吃。

### 2.5 最大上下文子图

以某个**焦点节点**为中心、沿边向外扩展、扩到 T 为止的那块连通区域。它就是大模型每一步实际阅读和推理的单位。其中：

- 真概念节点呈现**完整内容**；
- 合成枢纽呈现**摘要**（一处失真点）。

注意"完整"是相对的：低优先级的边（关联、出处、历史断言）会被 §2.7 的优先级截断挤掉，所以一块子图的完整性由**边优先级 + 预算**界定，不是无条件全量（见 §2.7、§3.2）。

### 2.6 别名词典（词法入口索引）

所有概念的 `name` 与 `aliases` 动态维护成一部**词典**，作为查询时定位入口的**倒排/词法索引**。它是 embedding-free 的入口手段，在命名实体上比向量更准（精确匹配不会"幻觉相似度"），代价是对"用户用描述/近义指代、字面没出现名称"的查询会漏——这部分由时序召回与顶层兜底覆盖（见 §5.1）。

实现要点：

- 它本质是**多模式串匹配**（在名称+别名集合上做 Aho-Corasick / 倒排索引），把用户输入分词后做全量匹配，找出所有命中的概念节点。中文场景下，把名称/别名注入分词器自定义词典，能让领域术语作为整体被切出。
- 词典必须随图变动**同步**：概念创建时登记、合并时取别名并集、改名/删除时更新。
- 它只负责"找出字面命中的候选"，不做语义判断；命中的候选是否真与 query 相关，仍由 §5 的语义遍历把关。

### 2.7 子图序列化与边优先级（喂给大模型的格式）

把一块最大上下文子图递给大模型时：

- **概念的相关说法：要喂。** 关系的含义在说法里（"喜欢红色"vs"讨厌红色"一翻答案就反），所以把当前焦点/邻居的**相关说法**带上——挑跟 query 相关、未 superseded 的那些。
- **边：只喂结构（无类型邻接 + 涌现方向），边本身无标签**。结构给模型的是这片知识的**形状**——谁是上位/类别、谁是细节、哪些聚成一簇、哪个是连两簇的桥；砍掉它递过去就是一堆散装解释。含义由说法补足，边只给形状。

**推荐格式**：层级用**缩进提纲**（枢纽一行、成员缩进其下），成环/跨簇关联用一句**交叉引用**标出（"X 另与 Y 相关"），关系含义把相关说法挂在对应概念下（"小明：喜欢红色@t1、喜欢蓝色@t2"）。几乎不费 token，且模型对提纲的理解远好于平铺列表。

**边优先级（在已结构性有界的视图里做辅助取舍）。** 一跳视图能否读完由结构保证（§3.1），优先级**不负责把视图压进预算**，只负责在这个已经有界的视图里排先后、并管"补充层要不要顺带塞"。按优先级贪心排序：

1. 与 query 直接相关的**说法**（答案材料）；
2. 通向相关区域的**导航/结构边**（骨架）；
3. 作为语境的**其它说法与邻接**；
4. **补充层（最低）**：出处边（事件 → 概念）、被 superseded 的历史说法、低置信说法——**一直存，但平时不喂**，只有预算有余量（数据稀疏）时才轮到它们。

**两个让补充层"被喂"的触发器**：① 预算有余量；② **query 本身就在问它**——出处/时序/审计类问题（"这是什么时候变的""依据是哪条"）会让出处边和历史断言从垫底**跳到最高优先级**，因为这时它们就是答案。所以优先级不是纯静态的，要让 query 调权。

**即使不喂原始出处边，也要在概念上留一个极小的出处/置信标注**（"当前、被复述 3 次、最近 t2"，或一个新鲜度/争议标志），让模型不读原始事实也能校准该信几分——别把信任刻度连同原始边一起删掉。

**分阶段**：导航时序列化从简（焦点 + 候选邻居 + 候选的关键说法 + 哪些是上位/枢纽），只在 §5.5 最终汇总推理时给完整提纲 + 交叉引用 + 相关说法。

## 3. 核心不变式与原理

### 3.1 局部容量不变式（硬约束）

> 对任意节点 N：`Tokens(N.content) + Σ Tokens(邻居.content)`（在被枢纽降扇出之后的一跳邻域上）`≤ T`。

含义：大模型"站"在任意节点上，把它和它的一跳邻域一次读进上下文，永远装得下、永远能看清下一步选择。一旦邻域超过 T，就触发降扇出（§4.3）。

**一个要点（结构优先，优先级辅助）**："一跳视图在一个窗口内可读"是**结构性的硬要求**，靠降扇出在 store 时保证——稠密的无类型边塌缩成星型，**说法过多**则归到中间聚合概念下并按需淘汰（§4.3），两者都让任意节点的直接扇出与说法量始终**有界、可读**。**边/说法优先级（§2.7）只是辅助**：在这个已经有界的视图里排个先后、决定补充层（出处/历史）要不要顺带塞、并随 query 调权。它**精修**视图，不**造出**可读性——可读性由结构保证，不能反过来靠截断兜底。

### 3.2 "完整"是按层完整，不是全量完整

`content` 表达的是这个概念**在这一层抽象上**的完整描述，更细的内容应当下沉成子概念。所以：

- 一个节点自身的 `content` 必须是**有界、可分解**的。若单个节点的内容本身就快撑满 T，需要把它的内容切成子概念（这是一个**内容分解**操作，和"降扇出"不是一回事，见 §8）。
- 完整性由**优先级**决定：与 query 相关的**说法**（高优先级）会被完整带上；**其它说法、出处边、历史说法**（低优先级）数量大、会被 Top-K / 优先级截断，**这些天生有损**。

### 3.3 控制规模的几种杠杆

- **无损重组（边）**：稠密区域里 k 个互相关联的概念会形成 O(k²) 条无类型边，塌缩成以枢纽为中心的星型后变成 O(k) 条——信息不丢，只是重组（§4.4）。它和去重、非对称涌现是同一个动作。
- **版本的归并与淘汰**：版本挂在属性节点上、不能像边那样塌缩；版本多的属性节点用两招控量——把同主题属性归到**中间聚合概念**下（§4.3），以及按**触发时间 + 置信度**淘汰过期/被矛盾的版本（§4.6）。
- **有损修剪（最后手段）**：删掉噪声边或冗余弱边（§4.7）。这是真在忘东西，排在无损重组之后，且有事件层兜底（删的是当前活跃图、不是历史）。

## 4. 图的演化（写入）

### 4.1 事实摄入

来一条原始事实 → 追加到事实层 → 抽取它提到的概念 → 为每个概念建立"事实 → 概念"的锚定边 → 把这些概念交给 §4.2 做放置。

### 4.2 新概念的放置，以及"它是否已在图里"的判断

这是写入最难的一步，难点在于：**没有全局索引，你无法证明一个概念"全局不存在"。** 所以不要去证明不存在，换一个提法。

原理：**一个概念的身份 ≈ 它的邻居。**

- 一个新概念总是和别的概念一起出现在某条事实里。把这条事实里**已经在图中**的那些共现概念当作**锚点**。
- 从锚点的邻域出发导航过去，让大模型阅读那块**完整内容**的局部子图，直接判断：这个概念是不是已经以某个节点 X 的形式存在于此？
  - 在 → 并入 X；
  - 不在 → 作为新节点插入，与这些锚点建无类型邻接边；事实带的关系含义记成新节点（或锚点）上的一条**说法**。
- 如果这条事实里的概念**一个都不在图里**（全新疆域），就没有锚点，退回到从顶层枢纽自顶向下导航，大概率落在空白区、直接开新区。

**不追求写入时唯一。** 漏判产生的重复，交给社区合并惰性兜底（§4.4、§4.5）。

### 4.3 扇出溢出 → 降扇出

当某节点一跳邻域超过 T，触发降扇出。把这批邻居整理出一层中间结构：

- **优先**：审查这批邻居里是否存在能收纳其余几个的上位概念，有就把它提上来当中介（带完整内容）。
- **兜底**：若全部平级，才造一个合成枢纽收纳一组，给它写一段摘要。

被提拔/新建的中介若自身也超 T，递归向下重复此过程。

**注意：降扇出针对边；版本另算。** 边一律无类型、都能塌缩。需要单独控量的有两处：① 一个实体挂了太多**属性节点**（小明的偏好、住址、经历…）→ 把同主题的归到**中间聚合概念**下（"小明的偏好""小明的基本信息"），让直接扇出降下来，再超就递归；② 单个属性节点的**版本列表**太长 → 靠 §4.6 的淘汰（触发时间+置信）控量。这样扇出与版本量**都是结构性地有界**，不靠 serve 时截断兜底（优先级截断只是最后的安全网）。

### 4.4 社区合并 → 涌现非对称 + 减少边

每当一片区域聚成社区（或新枢纽诞生），就**重新处理这片子图内的边**：把社区内那张稠密的双向邻接网塌缩成以枢纽为中心的星型——冗余对等边被消掉，枢纽 → 成员的有向边留下来，**非对称由此涌现**。（边一律无类型，都参与塌缩；含义在属性节点的版本里，不受影响。）

**为什么把"谁更一般"的判断放在这里**：判断一般性是这套系统里最难、最不稳的结构判断。放在摄入时（手上几乎没有上下文）必然飘；放在合并时，整个社区都在视野里（而且因为半窗口预算刚好整块装得下），大模型是在信息最全的时候才下这个结论。合并的逻辑必须**显式地让模型论证"为什么是这个当上位概念/枢纽"**，而不是随手挑一个；后续合并还能再修正它。

**合并的铁律：不能丢失语义。** 三条：① 幸存节点要**承接两边全部说法与邻接**；② 两边冲突（一个喜欢一个讨厌）要**并存 + 标时间/置信**，不能默默丢一边——那才叫丢语义；③ **没把握就不合并**。合并是破坏性、几乎不可逆的（两个概念塌成一个就拆不回来），而漏合并只是留个重复、良性可治（§4.5）。所以**宁可不合，不可错合**，拿不准就维持分开。

### 4.5 去重原理

**存在性检查和合并，用的是同一个信号——邻居集合的重合度。** 摄入时顺手查一次（§4.2），社区合并时再扫一次。一个真重复的概念，迟早会和它的孪生节点共享一堆邻居，在某次合并里撞上、被并掉。

**一个诚实的盲区**：若两个重复概念**完全没有共享邻居**（在互不相干的语境里各自被描述），导航找不到、邻居重合度永远为零，这个重复永远漏网。但这恰好是**危害最小**的情况——重复的真正坏处是割裂一个区域，而割裂只发生在它们彼此靠近时（正好能被逮住）；邻居完全不相交的两个节点，本就在扮演不同角色，留着不并几乎无害。**测不出的那种，刚好是可以不管的那种。**

### 4.6 更新与版本（属性节点上的版本化）

值随时间变化时（小明 t1 喜欢红、t2 喜欢蓝），**版本挂在属性节点上**：

- **实体与值节点都不复制**：小明、红色、蓝色各自永远是一个节点；变化记在「小明的爱好」这个属性节点的版本列表里。
- **事件层 = 不可变历史**：F1@t1、F2@t2 都在、永不删，它们本身就是版本历史 + 出处。
- **新事实冲突时，让 LLM 判"更新还是新增"**：关系是单值/排他（住址、最喜欢的颜色）→ 新版本当前、旧版本标 superseded（先不删）；多值/可累加（一般的喜好）→ 两条版本并存、只是时间戳不同。**安全默认是都留、各带时间戳，不在写入时销毁性覆盖。**
- **属性节点的版本列表是当前物化视图 + 极小的出处/置信标注**（当前/superseded、支撑事实 id、最近触发时间、"刚变过/有争议"标志），**不放全量历史**（保持有界）；要历史就查事件层。
- **查询时三种都能答**：当前（取未 superseded 的版本，多值再经 §5.4 仲裁）、历史（走事件层旧事实）、都给（并起来）。
- **把"当前适用置信"和"历史真值"分开**：蓝新→当前置信高，红旧→当前置信低（单值则近乎为零）；但"小明确实说过红"这件历史真值永远 100%、不衰减，只是不再当前。

**淘汰时间线（接 §4.8）**：被 superseded 的说法不立即删，先降为补充层、平时不喂（§2.7）；之后按 **触发时间 + 置信度** 打分——**长期不再被复述、且置信持续走低**的，最终从活跃概念图 GC 掉。事件层始终留底，所以"删"只是清活跃视图、可重放找回。注意触发淘汰的是"久未触发 + 低置信"两条同时满足，**不是单纯年纪大**（否则会淘汰冷门但正确的说法）。

### 4.7 边的删除与修剪

删边是在**忘掉关系**，和 §4.4 那种"只重组不丢信息"是两回事，所以排在最后：**不能合并时，先试无损重组（提拔现有概念 / 挂枢纽），删边是最后手段。**

- **能删的两类**：① 噪声/误连的"无关系边"——清理，但要**有把握**的判断（删边等于断一条路）；② 冗余弱边——但条件必须是**弱 且 结构冗余**（删掉它不让任何东西断开、也不显著拉长到达路径）。
- **桥再弱也不能删**：关联度低 ≠ 不重要，一条弱边可能是通往某片区域的**唯一桥**，删了那片在遍历里就到不了（同 §5.2 那个坑）。
- **怎么判强弱（不用每条边一次 LLM）**：置信度、被重复确认次数、新近度，加一个结构指标（是不是桥）。
- **事件层兜底让删边安全**：概念层删一条边并没真删知识——产生它的事实还在事件层。所以概念层删边是"**从当前活跃图清出去**"（可重放事件层找回），不是"从历史抹掉"。**只要不动事件层，概念层这层可以放心当垃圾回收来修剪。**

### 4.8 矛盾 vs 同义：两种"看起来重复/冲突"的不同归宿

这俩是两回事，别混为一谈：

- **同义（其实是同一个概念）**：无害。靠邻居重合度在某次社区合并里**惰性合并**（§4.5）。它们不会给出互相矛盾的当前值——本就是一回事，合了就一致。所以"保守合并、暂时漏合"在同义情形下确实没有代价。
- **矛盾（说法互不相容）**：不靠合并解决。被反驳的一方因为**得不到新事实复述、置信度持续走低**，按 触发时间 + 置信度（§4.6）最终从**活跃概念图里被 GC 掉**（§4.7），而产生它的事实仍留在事件层、没从历史抹掉。

两条要拧紧：

1. 置信度走低的触发应是"**被矛盾 + 不再被复述**"，**不是单纯"年纪大"**——否则会删掉冷门但正确的知识（呼应 §8 第 4 条：衰减作用在适用性，不作用在历史真值）。
2. 这里的"删除"始终是**活跃图的垃圾回收**，事件层永不动。所以一条曾经的说法永远查得到，只是不再当前。这也兜住了"漏合并的同义节点万一持有不同值"的担心：要么它们其实同义、合了就一致；要么它们其实矛盾、低置信那个会被 GC——不会留下永远无法调和的当前值。

### 4.9 知识修改流程（= 追加版本，从不原地覆盖）

1. **新事实进事件层**（只追加、永不改写）。
2. **定位实体与属性节点**：用 §4.2 的放置/导航找到涉及的实体（小明）和属性节点（小明的爱好）；属性节点不存在就新建，复用概念去重避免建出重复属性节点。
3. **判更新还是新增**（LLM，§4.6）：单值/排他 → 新版本设为当前、旧版本标 superseded；多值/可累加 → 并挂新版本。
4. **挂新版本**：把（值 + 时间 + 置信 + 出处）挂到属性节点的版本列表，更新该项的最近触发时间。
5. **维护**：被 superseded 的旧版本降补充层、按触发时间+置信走淘汰（§4.6）；若新信息暴露了重复概念 → 触发合并（§4.4/§4.5）；若属性节点版本/扇出超界 → 归并到子聚合或降扇出（§4.3）。

**要点：修改从不原地改写，永远是"加一条版本 + 给旧版本降级"。** 所以任意历史状态都能由事件层回放重建。

### 4.10 知识删除流程（两种，别混）

**A. 撤回 / 纠正（常规、可逆）**：来一条否定或更正事实（"小明其实不喜欢红"）走 §4.9 同一条路——旧版本被矛盾、置信走低、最终从活跃图 GC；事件层里"曾说喜欢红"和"后来纠正"两条都留，可回放。**这是默认的"删除知识"**，并没有真删任何东西。

**B. 硬删除 / 遗忘（合规、不可逆）**：唯一要动事件层的情况（隐私、法务要求彻底抹除）。tombstone 或物理清除**源事实** + 所有派生的版本/属性节点/边，并**重算**受影响概念（它们的支撑事实变了，物化视图要重建）。必须显式触发、留审计、不可回放。

两种都要做：

- **级联**：删一个值可能让属性节点空掉 → GC 该属性节点；删实体 → 级联其名下所有属性节点。
- **结构检查**：删点/边前查它**是不是桥**（§4.7），别把一片区域孤立、在遍历里变得不可达。

## 5. 查询：语义遍历（激活扩散）

查询不是机械的 BFS，而是**相关性引导的语义遍历**：每一步扩不扩展，都是一次大模型的语义判断。它本质是 embedding 出现之前那种图上原生的"激活扩散"，只是把判断换成了大模型阅读。

### 5.1 入口：生成种子节点集合（分级）

定位入口是整个查询的**上游难点**。MCS 用一套**分级入口**解决，从便宜到昂贵：

**第一级 · 关键词联想（词法召回）。** 用户输入经分词后，在别名词典（§2.6）上做**全量匹配**，命中的概念节点直接成为种子。这是最便宜、在命名实体上最准的通道。

**第二级 · 时序召回。** 从事实层按时间倒序取最近的一部分事实（事实节点带时间戳），这些事实**锚定的概念**也被召回为种子。这条通道补的是"用户大概在哪个近期语境里"，对追问、对延续近期话题的查询尤其有用。

**合并成种子集合，并受上下文限制约束。** 两条通道的结果合并成查询的种子节点集合。注意两点：

- **两条通道意图不同，不要平摊合并**：关键词召回 = "query 明确在问什么"（主），时序召回 = "近期语境"（辅）。对全新话题的查询，硬塞近期事实会污染种子，所以时序召回应作为次级、预算够才加，或仅用于消歧。
- **种子集合必须满足上下文限制（≤ T）**：命中过多时要排序裁剪。没有向量时的排序依据：匹配特异度（长/稀有别名 > 短常用词）、命中的 query 词数、节点新近度。装不下的，先以摘要形式作为初始前沿、交给 §5.2 的语义判断去优先化。

**第三级 · 顶点图兜底（仅当种子集为空时）。** 只有当词法召回与时序召回都为空时，才退回到从顶层枢纽自顶向下做语义导航（最贵、最模糊的那条路）。

一个贯穿的提醒：前两级召回是**高召回、低精确的候选生成**，不做语义判断。命中的种子是否真与 query 相关，仍由 §5.2 的语义把关——召回只是给遍历一个便宜的起点，省掉从根导航，不省语义把关。这是经典的"便宜召回 → 贵的语义重排"两段式。

### 5.2 扩展：每步一次语义判断

在一个节点上，**一次**大模型调用读完它整圈邻居（真概念是完整内容，枢纽是摘要），一次判完该往哪几个方向扩。没被选中的邻居不再各自花调用，是在这一次里被顺带评估掉的。所以调用轮次 ≈ "你决定钻进去的节点数"，而不是图里节点总数。

**最重要的判据**：每一步该问的不是"**这个节点本身和 query 相关吗**"，而是"**从这里扩出去，能不能通向 query 想要的东西**"。一个不起眼的桥/枢纽，自身内容泛泛，背后却可能挂着一整片高度相关的社区；只按局部相关性剪枝，会把藏在平淡连接点后面的相关内容整片切掉，卡在局部相关性的山头上。这个坑在**合成枢纽**上最凶——你恰恰是在只有摘要、信息最少的地方做"要不要钻进去"的判断。

判断时还应**带上已经捞到的内容**，让模型判的是"在我已有基础上，这个方向还**边际**有用吗"，避免反复拖进冗余的近邻。

**分阶段的序列化（承 §2.7）**：导航这一步和最终推理那一步需要的边信息不一样。导航时模型只是选个方向，给"焦点 + 候选邻居 + 候选的关键说法 + 哪些是上位/枢纽"就够了，邻居彼此之间的内部连边可有可无，喂得越精越好；只有 §5.5 最终汇总推理那一步关系结构最值钱，才把完整的提纲 + 交叉引用 + 相关说法都给上。两个阶段不必用同一种序列化。

### 5.3 累积与停止

遍历过程中把相关节点**存进结果累加器**：

- 广度上存 `summary`（因为存摘要，累加器才能装下远超一个窗口的节点）；
- 对最相关的那几个节点**回存完整内容**，让最终合成时细节不塌（两层结果）。

停止条件有二：

- **自然终点**：相关性前沿空了——没有前沿节点还过得了"通向答案"这关。（注意：连通图上不存在"遍历完整张图"这种终点，必须靠相关性来停。）
- **强制终点**：累积量**按 token 封顶**（不是按节点个数，因为摘要大小不一，而且结果最后还得整块塞回一个窗口做合成）。

### 5.4 仲裁：选出无矛盾的事实

遍历捞回的版本可能互相矛盾、或在一个粗属性节点上混挂不同侧面（小明的爱好：红@t1、蓝@t2、摇滚@t3）。**合成答案前，让 LLM 做一次仲裁**，按这次 query 选出一组**自洽**的事实：

- 问"现在 / 最新"→ 取当前、未 superseded 的；
- 问"曾经 / 历史"→ 全要，按时间铺开；
- 推理任务 → 取够用且互不矛盾的一组：同槽位互斥的（红 vs 蓝）只留一条，不同侧面的（蓝、摇滚）共存。

仲裁用版本的时间/置信和（§4.6 的）单值/多值知识来判。它是**查询时**的一致性解析，和写入时的 supersede/淘汰是两道关——写入尽力定，查询按这次问法最终定。它只读捞回的那几条版本，成本有界。

### 5.5 返回与追问加深

把仲裁后的内容合成一个答案返回给用户。若累积的摘要本身就超了一个窗口，合成这一步要做**分层摘要**。

用户若觉得不够，可以追问加深——这顺手躲掉了"该挖多深"这个无法提前调准的参数。要让它好用：

- **遍历必须可续跑**：把已访问集合、停下时的前沿、已返回的结果都存住，追问时接着扩、不重跑。
- **区分"要更多"和"要更深"**：更多 = 放宽相关性门槛、把同层更多邻居捞进来；更深 = 顺着某个分支往下钻、进到之前跳过的枢纽里。
- **追问那句话本身是新的相关性信号**：拿它给前沿重新排序、重新瞄准，而不是盲目顺原方向续。每次追问都在给这场激活扩散重新定向。

## 6. 贯穿全局的几条关键原则

1. **把最难的结构性判断推迟到信息最全的时刻**（合并时整个社区在视野里）再做。
2. **摘要质量是命门**：检索能不能找到一片区域、要不要钻进一个枢纽，都取决于（合成）枢纽的摘要有没有诚实预告里面有什么。真概念带全文则无此问题——所以**合成枢纽越少越好，能用真概念顶就顶**。
3. **判方向，不判局部相关**：看一步是否通向答案，而非节点自身相不相关。
4. **写入不保证唯一，靠惰性合并兜底**；存在性检查与合并是同一个信号（邻居重合度）在两个时刻。
5. **一个领域一个领域地建**：把自己保持在局部性最好、最接近教科书结构的区间里。
6. **一种无类型边 + 说法承载含义**：边只表达邻接、方向靠合并涌现；关系的"怎么"放在概念的说法里（自然语言、带时间/置信），不放在边上。减少规模先用无损重组、删边是最后手段且不删桥。
7. **事件层是历史与兜底**：版本化在断言不在节点；概念层是当前物化视图，superseded / 删边只动活跃图、不动事件层（随时可重放找回）。

## 7. 这套机制什么时候够用 / 什么时候发紧

- **强**：已能被人类整理成"可教结构"的领域（有教科书/本体/分类法）。人类已经替你把压缩和层级做好了。
- **弱**：跨多个领域的综合问题（相关节点散在互不靠近的区域，连通遍历捞不齐）；强语境依赖、矛盾常态、没有干净层级的知识。

**别在哲学层面争论"知识够不够简单"，让系统自己告诉你。** 在你每个具体领域上盯这几个信号，它们会指示赌注当下成不成立：

- 扇出/枢纽是不是频繁爆裂；
- 遍历跳数短不短；
- 相关性前沿是收敛还是越扩越散（散 = 局部性不够、交叉太密）；
- 用户是不是老追问（说明局部捞漏了散在别处的相关内容）。

## 8. 实现时仍需你拍板的开放点

1. **内容分解操作**：节点自身 `content` 过大时，怎么把它切成子概念。这与"降扇出"是两个不同操作，目前只定义了后者。
2. **关联完整性**：横向关联是 Top-K 有损的。是接受"只给最相关的几条"（多数场景够用），还是给关联也设计一套有界化机制。
3. **合并/重聚的触发与范围**：什么信号触发一次社区重聚（建议：枢纽溢出、低置信放置、摘要变含糊），每次重聚的范围控制在一个枢纽的子树内（≤ 半窗口，一趟调用能读完）。
4. **可选的记忆动力学**：若要引入"激活度/新近度"，务必把它当作**检索排序的信号**，不要当作**有效性的闸门**——本体里的概念不该因为最近没人问就变得不可信。衰减作用在"激活度"上，不作用在"概念是否成立"上。
5. **非确定性**：每步语义判断天然不确定，同一 query 两次可能走出不同路径。检索场景一般可接受，必要时可对关键路径做缓存。
6. **别名生成策略**：让大模型在概念创建时生成哪些别名、覆盖到什么程度（同义/缩写/口语/易错），直接决定走便宜词法入口的命中率。可考虑随使用反馈补录（用户用过、最终命中该概念的说法，回灌成别名）。
7. **种子排序与裁剪**：词法全量匹配命中超过上下文限制时，按什么排（匹配特异度 / query 词覆盖 / 新近度）裁到 ≤ T；以及关键词召回与时序召回两条通道的权重/优先级怎么定。
8. **词典与图的同步**：别名词典在概念合并（别名并集）、改名、删除时如何保持一致；一词多义在单领域内若仍出现，命中多个节点时如何处理。
9. **版本/属性节点的归并去冗**：同一属性节点会攒下很多措辞相近的版本，要不要、何时归并/去重（惰性、与概念去重同机制），以及版本列表的容量上限怎么定。
10. **单值 vs 多值关系的判定**：靠 LLM 按关系性质判（住址单值、喜好多值），决定新说法是 supersede 旧说法还是并存；判错要可被后续事实纠正。
11. **桥的识别成本**：修剪弱边前要判"它是不是桥"（割边/关节点），在大图上精确算代价高，可用增量/近似（如局部连通性检查）。
12. **淘汰打分**：触发时间与置信度怎么组合成淘汰分、阈值定在哪；务必保证"低置信 且 久未触发"才淘汰，置信不因单纯年龄而降。
