## 1. 算法概述

MCS（Maximum Context Subgraph，最大上下文子图）是一种把知识组织成图、并保证可检索子图有界的方法。它的核心是一条不变量：

> **图中任意节点的"活跃视图"，渲染成 token 后不超过阈值 `T`（`T` 是上下文窗口分给"查询内容"的那段预算，窗口划分见 §4）。**

- **活跃视图**：以某节点为中心、提供给 LLM 的内容——该节点本身，加上它的直接邻居（按优先级截断后）。
- **维持手段**（每次改图操作后即时把关，使每个节点的邻域有界）：
  - **聚类归纳**：节点邻域过大时，由 LLM 把同类邻居归纳到中间节点之下；
  - **优先级截断**：无法归纳的部分，按优先级保留高者；
  - **分层**：高基数的时间序列（事件）与有界核心分离（见 3.3）。
- **有界的对象是活跃 / 渲染视图，不是存储**：存储可无限增长，被保证的是"任一节点一跳邻域装得下"。
- **估算口径 == 渲染口径**：判断"是否超 T"的估算，必须与实际渲染逐字一致（同字段、同去重规则），否则会低估漏判、破坏不变量。

导航、归纳、查询都建立在这条不变量之上：因为每个节点的活跃视图都装得下，查询可以从入口节点沿图逐跳扩展，每一步都不会溢出上下文窗口。（窗口 `W` 具体怎么划分、`T` 从哪来，见 §4。）

## 2. 适用场景

MCS 是**通用**算法，不绑定具体语料。典型场景：

| 场景 | 形态 | 用 MCS 做什么 |
|------|------|--------------|
| **Wiki / 知识库** | 概念条目 + 交叉引用 | 跨条目事实关联、多跳检索、概念归纳 |
| **个人记忆 / Agent 记忆** | 对话流 / 笔记 | 命题沉淀、事件时间线、矛盾与时效管理 |
| **文档管理** | 文件 / 段落 | 原文保真回读、跨文档主题组织、出处追溯 |

**统一原则：领域差异是字段、关系与节点类的差异，不是为每个场景另造一套抽象。** 三场景共用同一份核心 schema（第 3 节），差异落在 `extensions` 软字段与插件层（第 6 节）。

## 3. 数据结构

图由**点**和**有向边**组成。点分 4 类，边的类型保持极简、谨慎增加。

### 3.1 点（Node）

**点分 4 类。** 类区分的是节点在算法里的**结构行为**，不引入领域身份（人物 / 地点 / 组织等是 `extensions` 软标签，不进核心）。

| 类 | 是什么 | 怎么产生 | 进核心组织（聚类） |
|----|--------|----------|:------------------:|
| **概念（concept）** | 一个名词（人 / 地点 / 对象 / 抽象概念……）；名词性事物的默认归属 | LLM 语义抽取 | 是 |
| **事实（fact / 命题）** | 关联若干节点的一句陈述，**谓词落在其 `content`**；可被事件背书、可与其他事实互斥 | LLM 语义抽取 | 是（聚类只重组不合并） |
| **事件（event）** | 时间轴上的一次发生（带时间、参与者等结构） | **规则**：按既定结构直接存，**不经 LLM** | 否（事件层，见 3.3） |
| **source** | 原始资料 / 文件 / 段落，**按类型区分** | **规则**：切分 + 分类，**不经 LLM**、保真不改写 | 否（叶子） |

**怎么产生：语义的归 LLM，结构的归规则**

- **概念 / 事实** 需从文本**语义抽取**——靠 LLM。
- **事件** 是原始输入**本身的结构**（一次发生的记录：时间、参与者等）——**输入须符合该结构**，系统按规则直接存，**不经 LLM**，也不做"这算不算事件"的判断。
- **source** 按**类型**（文件 / 网页 / 段落 / 对话记录……，由扩展名 / 格式 / 来源等规则判定）切分分类、保真存入，**不经 LLM**；类型决定怎么切分、大字段怎么处理。

**点字段：**

```
id            身份
name          名 / 标题（用于匹配与显示）
content       摘要 / 定义 / 命题说法（参与有界 token 估算）
node_class    概念 / 事实 / 事件 / source
extensions    开放属性（领域软标签、timestamp、置信度、帧 / 出处指针、**hub 标记**……）
```

**关于谓词落点**：开放谓词（喜欢 / 出生于 / 嫁给…）数量无限，**不做成边的类型或 label**（否则无限膨胀），而是**落在事实节点的 `content` 里**。"事实即一等节点"还带来两个边做不到的能力：事件可以**背书**某条具体事实、两条事实之间可以**互斥**（边连不了边，点才行）。

### 3.2 边（Edge）

**边是有向的（`source → target`），带一个 `type`。边的类型保持极简、谨慎增加——能不加就不加，未来按需慢慢加。**

当前类型集：

| `type` | 方向 | 性质 | 作用 |
|--------|------|------|------|
| **关联** | 存储为单向，两端邻接都索引、双向可达 | 结构（基础边） | 连接事实与其端点、概念间一般关联、以及聚类形成的"组织中心↔成员"；**无谓词语义** |
| **互斥** | 事实 ↔ 事实 | 语义（当前**唯一**语义类型） | 表示两条事实相互排斥：两事实之间连一条互斥边 |

- **关联**是图唯一的基础结构边。它既连接事实与端点、概念间关联，也承载聚类产生的"组织中心 ↔ 成员"关系。
- **没有独立的"层级"边**：组织层级不是要去建的边，而是**聚类的产物**——聚类把同类节点归到一个中心之下，这个父子结构用"关联边 + 中心节点的 `hub` 标记"表达即可（见 3.3），不需要单独的边类型。
- **语义关系类型当前只有"互斥"**；因果、背书等其它语义关系**暂不引入**，未来规范化后再逐步加（见第 6 节扩展纪律）。
- 开放谓词不进 `type`（落事实节点 content，见 3.1）。

**边字段：**

```
id            边身份（主键）
source_id     源节点 id
target_id     目标节点 id
type          关联 / 互斥
priority      优先级（派生值，为遗忘 / 入边·关联截断预留，默认 0.0）
extensions    边的元信息（置信度、时间……，默认不填充）
```

### 3.3 有界与分层（核心不变量如何成立）

把图分两层，**有界只对核心负责**：

| 层 | 内容 | 如何有界 |
|----|------|----------|
| **核心图** | 概念 + 事实 | 节点邻域过大时**聚类归纳**到组织中心之下 + **优先级截断** |
| **事件层** | 事件 | **单向绑入**核心（事件 → 事实 / 概念，仅事件侧索引，**核心侧不反查**）；不聚类；**按时间倒排截断** |

- **聚类是语义归纳，不是图算法**：某节点的关联邻域超 T 时，把该节点 + 其邻域成员一次喂给 LLM，归纳成若干语义内聚的社区并重组；**禁止用连通分量 / Louvain 等纯图聚类**。对事实**只重组不合并**（合并会抹掉事实身份，断掉事件背书与互斥）。
- **层级由聚类涌现**：重组会产生"组织中心"节点——某个概念 / 事实被选作一组同类节点的归属枢纽，成员经**关联边**挂到它之下。这棵由聚类长出来的树就是导航下钻的层级，**不依赖独立的层级边类型**。
- **关于 hub 标记**：系统给"组织中心"节点打一个 **`hub` 标记**，**纯粹用于反查与可观测**（看谁充当了中枢），**没有任何算法含义**，渲染给 LLM 时与普通节点无异。
- **载重规则**：核心任何节点都**不反查事件**——否则像"用户 / 我"这种连着海量事件的节点会把全部事件漏回核心、撑爆活跃视图。需要出处时走"按需的、有界的"事实 → 事件定向查（独立检索步，不进常驻视图）。
- **事件是"帧（timeline）相对"的**："我今天读了一本讲三年前故事的书"——只有"今天读书"在我的时间轴上；"三年前的故事"是知识，落核心事实（带叙述时间属性 + 出处），**不能**把"三年前"盖到我的时间轴。事件套事件靠**出处链跨帧**表达，每帧仍是一条扁平时间轴，不引入递归节点。

## 4. 上下文预算与划分

工作时，LLM 的上下文窗口 `W` 是稀缺资源。本节讲清三件事：`W` 怎么分、两个预算闸是什么、一次查询里哪些东西占预算、哪些不占。

### 4.1 窗口怎么分：W = S + T + R

上下文窗口 `W` 分成三段：

```
┌───────────────── 上下文窗口 W = S + T + R ─────────────────┐
│   S 系统窗口      │      T 查询窗口       │    R 结果窗口     │
│ (系统提示 + 指令) │ (喂给 LLM 看的图内容) │ (模型写回的结果)  │
└────────────────────────────────────────────────────────────┘
```

- **S 系统窗口**：系统提示 + 当前指令 / 任务，固定开销，通常不大。
- **T 查询窗口**：留给图内容的预算，也就是不变量里的 `T`——**任意节点的活跃视图 ≤ T**。
- **R 结果窗口**：模型写回结果的空间。MCS 内部调用产出的是**结构化决策**（抽取 / 聚类 / 选择），不是长篇作答；基于子图的作答推理是**下游消费方**的事，不占 MCS 内部预算（思考型模型的思考 token 也算 R）。
- **默认 `R = T`**：结果窗口一般与查询窗口等大；`S` 通常不大，所以 `T` 大致是去掉系统开销后的一半——这就是"≈ W/2"的由来，但精确说是 `W = S + T + R`、`R = T` 默认，比例可配置。

### 4.2 两个预算闸：`T` 与 `token_budget`

| 预算 | 管的是 | 在哪里把闸 |
|------|--------|-----------|
| **`T`（查询窗口）** | **单次**喂给 LLM 的图内容上限 | 守门聚类（写 / 连边 / 合并后，一个节点 + 其邻域 ≤ T）；查询期**每一跳** |
| **`token_budget`** | 一次查询**累积**答案的上限（`≤ T`，给积累区封顶、好给本轮活跃区留空间） | 查询期累积的答案子图 |

`T` 保证**任意一跳**都装得下——聚类裂变就是把任何超 `T` 的节点拉回 `T` 以内；`token_budget` 给跨跳累积的答案封顶，连同 `max_rounds` 让查询停下来。

### 4.3 一次查询的四个工作区

读取（read，BFS）的工作状态分四个区，**只有前两个进大模型、占用上面的预算**：

| 工作区 | 进大模型 | 内容 | 受哪个预算 |
|--------|:--------:|------|-----------|
| **积累区** | 是 | 已**确认相关**的节点 / 命题，逐轮累积，最终成答案子图 | `token_budget`（≤ T） |
| **活跃区** | 是 | **本轮**待 LLM 推理 / 筛选的候选（当前节点的活跃视图） | 占 `T` 的剩余（T − 积累区） |
| **visited** | 否 | 已处理节点的 id，去重、防重复遍历 | 不占（仅存 id） |
| **frontier** | 否 | BFS 待扩展的节点 id 队列 | 不占（仅存 id） |

- **每轮真正喂给 LLM 的** = `S`（指令）+ 查询 + 积累区（已确认）+ 活跃区（本轮候选）+ `R`（结果余量），整体 ≤ `W`；即 **积累区 + 活跃区 ≤ 查询窗口 `T`**。积累区逐轮变大、活跃区空间随之收缩，逼近 `token_budget` 即停。
- **把积累区也放进上下文**，是为了让 LLM 带着"已经确认了什么"来判断下一跳值不值得扩——代价是它占预算、必须封顶。
- **visited / frontier 只存轻量 id**，不渲染、不进上下文、不计 token。它们保证 BFS 不重复、不遗漏，却不消耗大模型"看"的预算——**簿记留在算法侧，只有"已确认 + 当前在看"才花 token**，这正是 MCS 能在大图上游走而上下文不爆的关键。
- **写入期更简单**，没有积累 / 活跃之分：抽取阶段一次喂一个输入分片，守门聚类阶段一次喂"一个节点 + 其邻域"（≤ T）。

> **口径一致**（见 §1）：以上每处"是否超预算"的判断，都须按**实际渲染**的字段与去重规则逐字估算——少算字段会漏判、直接破坏有界。

## 5. 算法流程

### 5.1 写入（ingest）

把原始输入入库（结构部分按规则、文本部分靠 LLM）、**对齐已有节点**、连边，写入期就把核心组织成有界。

```
                原始输入
                   │
                   ▼
  ① 规则入库（不经 LLM）
       • 整个输入（content）→ 记为一个事件节点（记录这一摄入行为、落用户时间轴）
       • 原始资料（source）  → 按类型切分分类，存为 source 节点（保真不改写）
       • content 内转述的过去事件（"三年前发生 X"）→ 不盖成时间轴事件，仍由 ③ 抽成带时间属性的事实
                   │
                   ▼
  ② 关联节点提取（复用 read 的检索能力）
       以输入的字面线索为种子 → 跑 read 的种子定位 + BFS
       把图中**已有的相关节点**（概念 / 事实）取进来，作为 ③ 的对齐上下文
                   │
                   ▼
  ③ LLM 语义抽取 + 对齐（带着 ② 取回的已有节点）
       • 名词       → 概念（与已有同义概念合并 / 复用，不重复建）
       • 关系命题   → 事实（谓词落 content；与已有事实对齐、判互斥）
       （记录概念 / 事实来自哪个 source / 事件，供 ④ 连背书边）
                   │
                   ▼
  ④ 连边
       • 事实  ──关联──>  端点概念
       • 事件  ──单向──>  事实 / 概念（背书·提及，按来源链接）
       • 事实  ──互斥──   事实
       • 无任何关联的孤儿  ──关联──>  虚拟根
                   │
                   ▼
  ⑤ 守门 + 聚类（仅核心：概念 / 事实，循环至收敛）—— 见下「细化」
                   │
                   ▼
  ⑥ 持久化
```

> **实现现状**：4 类节点 / 2 类边 / 双层 / 守门 / 互斥 / read-repair 均已落地；①④ 的
> **event/source 接入统一 `ingest`** 已实现——每次 `ingest` 把整个输入记为一个事件节点
> （落时间轴）+ 可选 source 切分，并对 ③ 抽出的概念 / 事实连背书边；`ingest` 入参
> `str | IngestInput`（`str` 经兼容垫片零改动）。事件 / source **只能经统一 `ingest` 产生**——
> 由内部原语 `_build_event_node` / `_build_source_nodes` + `_connect_endorsement_edges` 实现，无独立公开入口。

要点：
- **写入复用 read**：抽取之前先用输入线索把图中**已有的相关节点**检索回来（②），抽取 / 判定才能**对齐已有**——合并同义、链接到已有概念、判互斥，避免建出重复节点。这一步本质就是一次 read。
- **结构的归规则、语义的归 LLM**：事件 / source 规则入库、不经 LLM；LLM 只做概念 / 事实的抽取与对齐。这把"命题 vs 事件"的判断从 LLM 手里拿掉了（见第 7 节）。
- **只有孤儿挂根**：有关联可达的概念 / 事实不必挂虚拟根，避免根节点扁平化。

**守门 + 聚类（细化）**

```
  写入 / 连边 / 合并后，对每个受影响节点：
                │
                ▼
   ① 守门：估其层级视图（中心 content + 层级子节点，不含关系边）
                │
        ┌───────┴───────┐
      ≤ T            即将 > T
        │               │
        ▼               ▼
      放行    ╭─► ② 整窗单次喂：中心 + 全部层级子节点 → decide_hub
              │        │      （不变量保证此刻 ≤ T、一次装下）
              │        ▼
              │   ③ LLM 归纳成若干语义内聚社区（禁纯图聚类）
              │        │
              │        ▼
              │   ④ 社区三选一重组：
              │        合并同义(仅概念·事实不并) / 找关键概念 / 概括新概念
              │        │
              │        ▼
              │   ⑤ 重挂：成员 →关联→ 组织中心 H（打 hub 标记）
              │        允许重叠 · 无法归类留原中心 · 滤幻觉 id
              │        │
              │        ▼
              │   ⑥ 边吸收：扫 H 成员的父节点，若某 X 的子节点 ⊇ H 全部成员
              │        → 删 X→各成员、加 X→H（复用 hub、减边减扇出）
              │        │
              ╰── 新中心仍 > T？— 是（递归）
                       │ 否
                       ▼
                    收敛：处处一跳邻域 ≤ T
```

- **守门口径 = 层级视图（不含关系边）**：守门估的是受影响节点的**层级视图**（中心 content + 层级子节点），**不含关系边 token**——因为 `decide_hub` 只处理节点、聚不了关系边；关系边的有界由**查询期 Phase 2 按 `priority` 截断**兜。"估算 == 渲染"（铁律一）针对的是**查询视图（`select_facts`，含关系边）**的渲染估算，与守门的 fanout 估算是**两个不同对象**，不要混。
- **事实只重组不合并**（合并会断掉背书 / 互斥）；禁"信息碎片集合"这类空洞聚合标签。
- **边吸收（⑥）减边、不破界**：新组织中心 H 一生成就触发——某节点 X 的子节点 ⊇ H 全部成员时，把 X→各成员换成 X→H，复用 H。X→H 替掉多条边后 **X 扇出变小**（不会超 T，无需再守门）；扫描只查 **H 成员的父节点**（候选），不必真遍历全图。

**图质量如何收敛（概念去重 / 合并）**

收敛**由读写共同触发、且不止一次机会**——一次没并上，后续还能并；最终兜底是后台维护：

| 时机 | 机制 | 并掉什么 |
|------|------|----------|
| **创建时**（写入 ②③） | 检索已有 → 对齐：同名复用、同义合并 | 检索召回到的重复，当场不建 / 合并 |
| **之后被触及时**（再写到附近，或**读取碰到**） | 工作集里冒出重名 / 同义节点 → 同名当场字面识别、同义搭"正在看这些节点"的 LLM 顺带判 → 合并或挂起 | 创建时漏检、日后被写 / 读碰到的重复（高频、近乎免费） |
| **聚类时**（守门 ⑤「合并同义」） | 两重复落入同一邻域、该节点裂变 → 合并 | 共享邻居、被一起聚类的重复 |
| **后台维护**（可选扩展，见 6.2） | 周期扫描 embedding / 别名相近的节点 → 合并 | 既没被触及、也没被聚类的**长尾残留** |

- **一次不成功不要紧，读也算一次机会（read-repair）**：重复一旦建出，之后每次被**写**或**读**"碰到"、或被聚类，都是再合并的机会——读取时**重名靠字面当场可见（零成本）**，同义搭 ③ 已在看这些节点的 LLM 顺带判出。收敛是**持续、最终**的，不是一锤定音。
- **合并要过守门**：合并把两节点邻域并到一起、可能撑过 `T`，所以**合并和写入一样要过守门**（见上「守门 + 聚类」——守门挂在"改变邻域的操作"上，不专属写入）。合并真重复时邻域多半高度重叠、并完仍 ≤ T，守门**当场廉价通过**（仅 token 估算、无 LLM）；只有并完**超 T（需聚类）或需 LLM 消歧**的，才**挂起交写 / 维护**，不在读路径同步跑 `decide_hub`。
- **保证最终收敛靠后台维护**：纯靠读写触发，"建出后再没被碰过、也没触发聚类"的冷门重复会**长期残留**；要保证一定收敛，需**周期性维护扫描**（可选扩展，有后台算力成本）把长尾并掉。
- **合并的代价**：靠 LLM 判同义 / 消歧（**同名 ≠ 同义**，如"苹果"公司 / 水果），有错合 / 漏分风险。
- **事实不走合并**：去重只针对**概念**；同一事实靠 ③ 按"同主·同宾·同说法"对齐去重（合并会断背书 / 互斥）。

### 5.2 查询（read）

种子定位后在核心做 BFS；积累区 / 活跃区进 LLM，visited / frontier 只做簿记；事件默认不进、按需取。

```
              查询
               │
               ▼
  ① 种子定位  顶层种子 / 切词·字面匹配名·别名 / embedding / 热门事件反查（**未来**，暂未实现、不在 `unified-graph-schema` change 内）
               │           → 节点仲裁 → 入口种子 → frontier
               ▼
  ╭─► ② 从 frontier 取一节点，渲染活跃视图（关联邻居，≤ 剩余 T）→ 活跃区
  │        │
  │        ▼
  │   ③ LLM 双角色筛选活跃区命题 / 邻居：`结果` → 积累区（吃 T）；
  │        │   `探索` → 下一轮 frontier（不吃 T）；两者入双方
  │        ▼
  │   ④ 选中者（任一角色）入 visited 去重；事实边端点随边角色补入
  │        │
  ╰── 继续：未达 max_rounds 且未超 token_budget（frontier 另受 max_frontier_nodes 阀兜）
           │ 否则 ↓
           ▼
  ⑤ 需出处 / 证据？— 是 → 事实 → 事件 定向查（默认不带事件）
           │
           ▼
  ⑥ 后处理（重排 / 裁剪）→ 返回子图（= 积累区）

  进 LLM·吃 T 预算： 积累区 + 活跃区      只存 id·不进 LLM： visited + frontier
```

- **探索口径 ≠ 结果口径（双角色解耦）**：③ 的 `select_facts` 同一次调用输出双角色——`探索`（宽召回，进 frontier、仅驱动 BFS 下一跳、不进 LLM、不吃 T）与 `结果`（精筛，进积累区、吃 T、是返回集）**成员可不同**。frontier 用完即弃（不返回），避免"宽召回直接撑大进 LLM 的积累区"。写侧 `select_facts_write`（窄召回）输出纯编号，`parse` 归一为"两者"，写路径逐字等价。
- **写入的 ② 关联节点提取复用的就是 ①–④ 这套**（种子定位 + BFS），见 §5.1。
- **读也顺带收敛（read-repair）**：工作集里撞见重名 / 同义节点 → 同名字面当场识别、同义搭 ③ 的 LLM 顺带判 → 合并。**合并产生的节点同样过守门**（多半廉价通过；并完超 T / 需消歧的挂起交写 / 维护，不在读路径同步聚类）。详见 §5.1 收敛表。
- **`purpose` 控制可见性**：导航隐藏 source 大字段（保 ≤ T），取回原文时才返回大字段。

## 6. 扩展机制：什么是设计，什么是扩展

清晰区分**核心设计**（稳定契约，改动须经评审）与**扩展内容**（可插拔、按需增长）。

### 6.1 核心设计（稳定契约，不轻易动）

- **4 类节点**：概念 / 事实 / 事件 / source。
- **产生方式**：概念 / 事实靠 LLM 语义抽取；事件 / source 由**规则**入库、不经 LLM（事件按既定结构直接存、source 按类型切分分类）。
- **有向边 + 极简类型**：基础结构边 `关联`，语义当前仅 `互斥`（无独立层级边——层级由聚类涌现）。
- **hub 仅为标记**（反查用，无算法含义）。
- **核心不变量**：任意节点活跃视图 ≤ T；估算口径 == 渲染口径。
- **分层与有界机制**：核心（聚类 + 优先级截断）/ 事件层（单向绑入 + 时间倒排截断）；核心不反查事件。
- **守门 = 改图即把关**：写入 / 连边 / 合并 / 读修复后都过守门，超 T 必聚类裂变；新组织中心生成时做**边吸收**（复用 hub、减边）。
- **谓词落点**：开放谓词进事实节点 content，不进边。
- **上下文预算划分**（第 4 节）：`W = S + T + R` 的窗口划分（`R = T` 默认）、两级预算（`T` / `token_budget`）。
- **ingest / read 骨架**（第 5 节流程）。

### 6.2 扩展内容（可插拔、按需增长）

- **领域语义** → 事实节点 `content`（开放谓词）+ `extensions` 软字段（领域子类型、领域属性）。
- **新增边类型** → 谨慎、登记制增加（如未来的 `因果` / `背书`）；当前刻意只保留 `互斥`。能用事实节点表达的，优先用事实节点，不轻易加边类型。
- **插件**：抽取 / 判定 / 重排 / 索引 / 持久化等阶段经插件接口扩展，不改核心 schema。
- **优先级与遗忘**：`priority` 评分、时间衰减 / 遗忘为派生机制，可独立演进（核心有界口径不依赖其具体实现）。

### 6.3 扩展纪律

- **能不加就不加**：边类型尤其谨慎——开放谓词一律落事实节点，避免类型集膨胀回退成无界 label。
- **加必登记**：任何新类型 / 新字段约定须登记，可索引、可解释。
- **不破不变量**：任何扩展不得违反"活跃视图 ≤ T"与"估算 == 渲染"。

## 7. 已知边界与代价

- **抽取质量（主要 LLM 依赖）**：LLM 从文本抽概念 / 事实，抽得准不准（漏抽、概念 vs 事实误判、谓词归纳偏差）直接影响图质量，须用样本量化。**注**：事件 / source 由规则入库、不经 LLM，所以**不存在"命题 vs 事件"误判、也不会把转述当亲历污染时间轴**——这层脆性被规则化消除了（文本里转述的"三年前发生 X"会被抽成带时间属性的**事实**，而非盖到用户时间轴的事件）。
- **去重依赖检索召回 + 同名歧义**：异名同义概念若写入期检索（§5.1 ②）没召回，会暂存重复、待聚类「合并同义」事后兜底——收敛是**最终的、非单次**。另外**同名未必同义**（如"苹果"公司 / 水果），消歧靠 LLM、有误判（错合 / 漏分）风险。
- **事件全量查的新近偏置**：时间倒排救最近、丢久远；靠把关键事件的稳定内容**蒸馏成核心事实**来兜。
- **入边 / 关联截断丢冷门**：高度数概念（热门对象）或高参与事件的连接会被优先级截断，靠 `priority` / 蒸馏缓解。
- **n 元互斥**：当前互斥是两事实间一条边；多于两条事实互相排斥时为两两连边，规模大时偏冗余——未来可优化，暂不处理。
